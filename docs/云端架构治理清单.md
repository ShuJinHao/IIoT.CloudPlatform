# 云端架构治理清单

本文档是 `IIoT.CloudPlatform` 安全与架构治理计划入口。最新状态以本文和滚动复盘 `改动复盘与规则沉淀.md` 为准。当前计划只覆盖 Cloud，不授权修改 `IIoT.EdgeClient`、`AICopilot` 或工作区顶层 `docs/总规则.md`。

## 0. 执行口径

- 当前生产部署必须保持内网受控 HTTP，不启用 HTTPS/TLS/443/HSTS/HTTPS redirection。
- TLS/HTTPS 不再作为当前批次安全红线；证书采购、证书签发、证书轮换和 HTTPS 改造不进入本计划。
- 未经用户明确批准证书预算、证书来源、变更窗口和允许修改目录，不得新增 nginx 443 listener、TLS 证书挂载、HSTS、`UseHttpsRedirection` 或强制 HTTPS 配置。
- HTTP 只允许用于受控内网、VPN 或等价专线环境；如果要公网暴露，必须先暂停并重新评审传输安全方案。
- 因 HTTP 不提供传输加密，本计划优先收紧补偿控制：强 secret、短期 JWT、bootstrap secret 恒必需、限流、最小权限、runner/SSH 权限收敛、日志脱敏、部署前 fail-fast 检查和网络边界控制。
- 没有进入本清单的审计项视为未处理；每项完成时必须更新状态、影响面、验证命令和规则沉淀位置。
- 代码改动完成前必须更新 `docs/改动复盘与规则沉淀.md`；形成长期规则时必须写入 `docs/云端规则.md` 或专题契约。

## 1. 当前已确认事实

- `deploy/nginx/nginx.conf` 当前监听容器内 `8080`，已有 CSP、`X-Content-Type-Options`、`X-Frame-Options`、`Referrer-Policy`、限流、上传体积限制和目录 listing 禁止。
- `deploy/docker-compose.prod.yml` 当前对外仍映射 `${GATEWAY_HTTP_PORT:-80}`，容器内部 nginx/gateway/HttpApi/web 均走 `8080`，保持 HTTP-only 口径但不再靠容器内低端口。
- `ConfigurationGuardTests` 已扩展 HTTP-only 守卫；后续不能把 HTTP-only 本身翻成失败项，只能补强 HTTP 基线下的安全补偿控制。
- `deploy/.env.example` 已移除 `10.98.90.154`、`change-me-*` 和 `ALLOW_INTRANET_HTTP_OIDC=true` 默认值，改用内网示例域名和不可上线占位 secret；`pre-deploy-check.sh` 已通过 `release-common.sh` 拦截模板值。

- JWT bearer 当前已启用 issuer、audience、lifetime、signing key 校验；本计划不全局禁止 `ReadJwtToken`，只禁止未验签 token 参与认证/授权判断。
- `CurrentUser` 已读取全部 role claims，Admin bypass 已统一使用 `Roles.Contains(SystemRoles.Admin)`。
- Human 登录和 Edge operator 登录已执行 Identity lockout：失败计数、锁定判断和成功清零。
- `DeleteCascadeAsync` 已在 EF 事务中显式删除并写审计，影响查询和批量删除已覆盖 `EdgeHostPlcRuntimeState`；旧 `EdgeHost` / `EdgeHostPlcBinding` 手工配置表不再作为运行时依赖。
- AI Read `/api/v1/ai/read/production-records` 已固化为唯一 AI 生产数据只读入口，生产记录查询已做公共列和 schema 字段过滤，并由 `docs/AI只读接口契约.md` 和 ConfigurationGuardTests 守卫。
- EdgeHost PLC runtime state 已固化为 Edge 设备身份写入、Human 只读和专用 store 方向，并由 `docs/Edge上传与PLC状态接口契约.md` 和 ConfigurationGuardTests 守卫。
- Cloud Web 未登录页已删除固定设备数、产量、合格率、告警、趋势和同步状态；登录后生产概览只允许根据真实 Human API 进入 `Loading / Empty / Error / Ready`，不再用固定趋势或固定班组填充空数据。

## 5.3 Redis 分布式锁租约完整性

### C-22 续租失败必须使写链路 fail closed

- 严重级：BLOCKER
- 状态：4A 与 4B 已实施；Redis 租约合同、有界操作、所有权丢失传播、HTTP 稳定语义和确定性测试已闭环，ClientRelease Host/Plugin 也已共用完整 handler 外层的 Redis 租约，不再使用本地文件锁。
- 审计纠偏：原 Redis 实现的 GUID owner token、`SET NX + TTL` 和 compare-renew / compare-delete Lua 本身正确；真实缺陷是续租返回 0/超时/异常后只记 warning，应用 handler 仍可继续并返回成功，且 Redis 调用和续租任务收口没有显式硬超时。
- 合同收口：`IDistributedLockService.AcquireAsync` 只返回 `IDistributedLockLease`，租约暴露 `OwnershipLost` 并实现 `IAsyncDisposable`。旧 `Task<IAsyncDisposable>` 合同物理删除，不保留重载、adapter 或 feature flag。
- 失效语义：续租 0/异常/超时会一次性触发 `OwnershipLost` 并结束循环。`DistributedLockBehavior` 将调用方 token 与租约 token 联结；调用方取消保留 `OperationCanceledException`，租约丢失无论 handler 是否响应取消都转为 `DistributedLockOwnershipLostException`。解锁失败不覆盖 handler 原始异常，成功路径上的解锁失败不得静默。
- 时间与原子性：Redis acquire/renew/unlock 和 renewal shutdown 均由显式配置的 `WaitAsync` 上限约束，并通过 `ValidateOnStart` 保证 `cadence + 2 * operation timeout < lease`。`acquireTimeout` 是覆盖 Redis 尝试和 retry wait 的单调总预算；单次 Redis 上限取剩余总预算与 operation timeout 较小值，零只执行一次不等待尝试。取消或超时后晚到的 acquire 成功会用原 token compare-delete；后台晚到任务均观测 fault。owner token 和两个 Lua 原子脚本保留不变。
- 错误与脱敏：竞争/等待截止映射 `409`，Redis 不可用、操作超时和所有权丢失映射 `503`。结构化日志只记 EventId、稳定 operation/reason 和异常类型，不记锁 key、Redis 原始 message 或 stack。Redis 故障不得降级为本地锁或无锁继续写。
- 测试和验收：
  - ServiceLayer 按 Redis lease service、Redis primitive 原子脚本、MediatR behavior、options/DI 四类拆分，公共可控 primitive/delay/logger 只保留一份 test-support。使用可控 `TaskCompletionSource` 覆盖竞争、零时不等待尝试、短总预算不被长 operation timeout 穿透、retry wait 总预算、`409/503` 分界、调用方取消、acquire/renew/unlock 超时与晚到 fault/成功、续租 0/异常、owner token 不匹配、阻塞/抛异常的取消回调、Dispose 幂等/硬超时、handler 响应/忽略租约取消、原异常优先级与启动配置验证；不使用猜测性 `Task.Delay`。
  - EndToEnd 直接执行全局异常处理器，冻结 `409/503` 和响应脱敏；宿主 appsettings 守卫新增的操作/收口超时字段。
- 4B 收口：ClientRelease 本地文件锁、nullable/async 会话入口、重复 409 分支和相关测试死引用已物理删除；未保留 adapter、fallback 或 feature flag。

### C-23 ClientRelease 共享发布租约边界

- 严重级：BLOCKER
- 状态：4B 已实施；Host/Plugin marker 命令通过同一稳定 Redis 资源和非零 `5s` 总预算获取租约，租约覆盖 body 读取、staging、验证、落盘、数据库提交、保留/清理和审计的完整 handler。
- 顺序和错误：`AuthorizationBehavior` 必须位于 `DistributedLockBehavior` 之前；未授权时不 acquire、不进 handler、不读 body。Host/Plugin 竞争沿用 4A 稳定 `409`，Redis 不可用或所有权丢失沿用 `503`；不恢复 Controller 业务错误特判。
- 会话合同：`ClientReleaseUploadCoordinator.Begin` 是唯一、同步、非空入口；`ClientReleaseUploadSession` 只实现 `IDisposable`，同步删除 staging。不得恢复 `TryBeginAsync`、`ValueTask` 伪异步、`GC.SuppressFinalize`、文件锁持有或并发空分支。
- 取消语义：handler 必须使用 Behavior 联结 token，并在上传、解包、验证、文件发布、数据库提交和提交后清理相位传播取消。Host `commitPointReached` 与 Plugin `databaseSaved` 保持自然提交点：提交前回滚本次产物，提交后不回滚已提交产物。
- 范围边界：4B 只迁移互斥边界，不新增 fingerprint、commit probe、Unknown 异常、目录复用、幂等表或数据库 migration。提交歧义必须在后续 4C 使用 fresh DbContext/独立基础设施读端设计；同一 scoped repository 的查询不是独立提交确认。
- 测试：使用共享 singleton 可控租约 fake 和两个 DI scope 覆盖真异步 acquire、同资源/`5s` 预算、Host/Plugin 竞争与释放后重入；行为组合测试覆盖 permission 在 lease/body 之前、ownership loss/caller cancellation 和 staging 清理；端到端门禁直接调用生产唯一 MediatR 注册 helper 检查 Authorization/DistributedLock 顺序。不连接真实生产 Redis，不使用猜测性 `Task.Delay` 或源码字符串断言。

## 5.4 ClientRelease 提交歧义前置隔离

### C-24 审计持久化不得复用业务 DbContext（4C1）

- 严重级：HIGH
- 状态：4C1 已实施；本批只隔离审计持久化，是后续 ClientRelease commit ambiguity 治理的前置条件，不代表已完成提交查证或同包幂等重试。
- 真实问题：`EfAuditTrailService` 原先与业务仓储共用请求 scoped `IIoTDbContext`，调用 `SaveChangesAsync` 时会顺带提交该 context 中所有 pending changes；审计失败后只 Detach 审计记录也无法证明业务 context 仍可用。对提交结果不确定的 ClientRelease 链路，继续使用同一 context 写审计会放大误判。
- 实施边界：`IAuditTrailService` 合同和 Scoped DI 保持不变；`EfAuditTrailService` 改为注入已验证的 `DbContextOptions<IIoTDbContext>`，每次调用创建、`await dispose` 一个独立 context，只 Add/保存当次 `AuditTrailRecord`。旧 Detach 补丁物理删除；不新增 ClientRelease 审计 adapter、service locator、migration、fallback 或兼容双轨。
- 取消和日志：调用方 token 已取消时的 `OperationCanceledException` 继续向上抛出，不得被 best-effort catch 吞掉。其他 context 创建/保存/释放异常仍 best-effort，但只记录稳定 `PersistenceFailed` EventId 与异常类型名，不传 exception 对象，不记原始 message、target id/path 或 secret。
- 测试：ServiceLayer 使用 SQLite in-memory 真实关系型 EF provider 和 `SaveChangesInterceptor`，验证业务 context 存在 pending aggregate 时审计只落审计表、审计失败不污染/清空业务 context、连续两次审计使用两个 context、caller cancellation 传播以及完整记录正常落库。不使用 InMemory provider 冒充关系型 SaveChanges 语义，不使用源码字符串断言。
- 未解决项：Host/Plugin `commitPointReached/databaseSaved` 的 ACK-lost 误判、fresh commit probe、Host manifest/Velopack artifact hash 语义、lease-loss 下的条件式文件回滚、exact retry 的 `200/409/503` 和 Unknown 审计语义均保留到后续独立可回滚批次；4C1 不允许顺手修改。
