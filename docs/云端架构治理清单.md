# 云端架构治理清单

本文档是 `IIoT.CloudPlatform` 安全与架构治理计划入口。最新状态以本文和滚动复盘 `改动复盘与规则沉淀.md` 为准。当前计划只覆盖 Cloud，不授权修改 `IIoT.EdgeClient`、`AICopilot` 或工作区顶层 `docs/总规则.md`。

> 现行测试执行只认本文置顶的“Cloud required 测试执行事实”和 `CLOUD-TEST-ARCH-001`：18 个物理 runner / 752 case，其中 required 16 / 688，通过 inventory 无 filter 对账。后续 C-xx 与批次记录中出现的旧测试桶、定向 `--filter`、旧计数和一次性声明迁移账本均是历史阶段证据，不得作为现行 CI/本地验收门禁。

## 0. 执行口径

- 当前生产部署必须保持内网受控 HTTP，不启用 HTTPS/TLS/443/HSTS/HTTPS redirection。
- TLS/HTTPS 不再作为当前批次安全红线；证书采购、证书签发、证书轮换和 HTTPS 改造不进入本计划。
- 未经用户明确批准证书预算、证书来源、变更窗口和允许修改目录，不得新增 nginx 443 listener、TLS 证书挂载、HSTS、`UseHttpsRedirection` 或强制 HTTPS 配置。
- 三项目总计划完成度只由工作区唯一活动入口 `../../docs/三项目测试架构治理总计划.md` 维护，本 Cloud 清单不再复制阶段百分比；Cloud 仓内测试治理状态以本文的 required 事实、`CLOUD-ARCH-001`、`CLOUD-TEST-ARCH-001` 和当前 PR 证据为准。
- HTTP 只允许用于受控内网、VPN 或等价专线环境；如果要公网暴露，必须先暂停并重新评审传输安全方案。
- 因 HTTP 不提供传输加密，本计划优先收紧补偿控制：强 secret、短期 JWT、bootstrap secret 恒必需、限流、最小权限、runner/SSH 权限收敛、日志脱敏、部署前 fail-fast 检查和网络边界控制。
- 没有进入本清单的审计项视为未处理；每项完成时必须更新状态、影响面、验证命令和规则沉淀位置。
- 代码改动完成前必须更新 `docs/改动复盘与规则沉淀.md`；形成长期规则时必须写入 `docs/云端规则.md` 或专题契约。

## 5.3 Redis 分布式锁租约完整性

### C-22 续租失败必须使写链路 fail closed

- 严重级：BLOCKER
- 状态：4A 与 4B 已实施；Redis 租约合同、有界操作、所有权丢失传播、HTTP 稳定语义和确定性测试已闭环，ClientRelease Host/Plugin 也已共用完整 handler 外层的 Redis 租约，不再使用本地文件锁。
- 审计纠偏：原 Redis 实现的 GUID owner token、`SET NX + TTL` 和 compare-renew / compare-delete Lua 本身正确；真实缺陷是续租返回 0/超时/异常后只记 warning，应用 handler 仍可继续并返回成功，且 Redis 调用和续租任务收口没有显式硬超时。
- 合同收口：`IDistributedLockService.AcquireAsync` 只返回 `IDistributedLockLease`，租约暴露 `OwnershipLost` 并实现 `IAsyncDisposable`。旧 `Task<IAsyncDisposable>` 合同物理删除，不保留重载、adapter 或 feature flag。
- 失效语义：续租 0/异常/超时会一次性触发 `OwnershipLost` 并结束循环。`DistributedLockBehavior` 将调用方 token 与租约 token 联结；调用方取消保留 `OperationCanceledException`，租约丢失无论 handler 是否响应取消都转为 `DistributedLockOwnershipLostException`。解锁失败不覆盖 handler 原始异常，成功路径上的解锁失败不得静默。
- 时间与原子性：Redis acquire/renew/unlock 和 renewal shutdown 均由显式配置的 `WaitAsync` 上限约束，并通过 `ValidateOnStart` 保证 `cadence + 2 * operation timeout < lease`。`acquireTimeout` 是覆盖 Redis 尝试和 retry wait 的单调总预算；单次 Redis 上限取剩余总预算与 operation timeout 较小值，零只执行一次不等待尝试。取消或超时后晚到的 acquire 成功会用原 token compare-delete；后台晚到任务均观测 fault。owner token 和两个 Lua 原子脚本保留不变。
- 错误与脱敏：竞争/等待截止映射 `409`，Redis 不可用、操作超时和所有权丢失映射 `503`。结构化日志只记 EventId、稳定 operation/reason 和异常类型，不记锁 key、Redis 原始 message 或 stack。Redis 故障不得降级为本地锁或无锁继续写。
- 测试和验收：
  - ServiceLayer 按 Redis lease service、Redis primitive 原子脚本、MediatR behavior、options/DI 四类拆分，公共可控 primitive/delay/logger 只保留一份 test-support。使用可控 `TaskCompletionSource` 覆盖竞争、零时不等待尝试、短总预算不被长 operation timeout 穿透、retry wait 总预算、`409/503` 分界、调用方取消、acquire/renew/unlock 超时与晚到 fault/成功、续租 0/异常、owner token 不匹配、阻塞/抛异常的取消回调、Dispose 幂等/硬超时、handler 响应/忽略租约取消、原异常优先级与启动配置验证；不使用猜测性 `Task.Delay`。
  - EndToEnd 直接执行全局异常处理器，冻结 `409/503` 和响应脱敏；宿主 appsettings 守卫新增的操作/收口超时字段。
- 4B 收口：ClientRelease 本地文件锁、nullable/async 会话入口、重复 409 分支和相关测试死引用已物理删除；未保留 adapter、fallback 或 feature flag。

- 严重级：BLOCKER
