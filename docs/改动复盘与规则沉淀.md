# 改动复盘与规则沉淀

本文档是 `IIoT.CloudPlatform` 的项目滚动复盘入口，最新记录放在最前。每次代码改动完成前必须更新本文件，并判断是否需要把长期结论沉淀到 `docs/云端规则.md`、专题契约或工作区 `../docs/总规则.md` / `../docs/历史核心记录.md`。

默认只更新本滚动文档，不为每个任务新增单独流水文档。只有形成可长期复用的业务规则、技术契约、部署口径或验收口径时，才新增专题文档。

没有形成长期规则时，复盘记录必须写明固定结论“无新增长期规则”及原因。

## 2026-06-30 客户端状态拆分、发布包删除和设备删除级联收口

- 改动范围：Cloud 运行心跳实体/迁移/API/查询、客户端发布 catalog 删除文件命令和 Human API、设备级联删除影响统计/缓存失效、Vue 设备删除确认和客户端发布页面状态列/操作列、Cloud 后端/前端测试与规则文档。同步修改 `IIoT.EdgeClient` 仅限运行心跳调用方契约，已获用户跨项目授权。
- 改动原因：设备安装状态把“版本上报超过 24 小时”误读成未启动，会误导管理员；发布 catalog 缺少管理员物理删除包能力，废弃插件包长期留存误导人；设备删除后普通用户设备授权缓存未清理；删除弹窗要求重复输入设备名称体验差。
- 影响面：新增 `edge_device_runtime_heartbeats` 表和 `POST /api/v1/edge/runtime-heartbeats`；设备客户端状态页拆成软件状态、最后运行心跳、安装状态、当前版本、版本上报时间和问题；管理员可在发布管理中删除未在用发布包文件；设备删除会级联运行心跳并清理受影响员工授权缓存。未把运行心跳伪装成 Cloud/MES 上传状态，Cloud 问题列没有真实来源时保持空值。
- 关键实现：运行心跳按 `device_id + client_code` upsert 并校验身份链；安装状态只看版本/catalog 差异，软件状态只看运行心跳；发布包在用判定只看当前版本快照是否等于目标版本，不加时间窗口；物理删除路径由服务端 release 元数据推导，限制在 `/edge-updates` 根内并拒绝 `..`、符号链接/重解析点逃逸；删除前审计文件清单；设备删除前捕获受影响员工 ID，删除后清理 `DeviceAccessesByUser`。
- 规则提炼：形成长期规则，已写入工作区 `docs/总规则.md` 和 Cloud `docs/云端规则.md`：版本上报、运行心跳、上传同步是不同事实源；运行心跳不得替代安装状态或上传状态；发布包物理删除必须用 `ClientRelease.Manage`、当前快照在用保护、服务端路径推导和文件清单审计；设备删除必须清理受影响员工授权缓存。

## 2026-06-30 Cloud 本机标准发布流程重写

- 改动范围：Cloud 部署文档、`deploy/scripts/build-and-push.sh`、`deploy/scripts/local-release.sh`、`cloud-image` / `cloud-deploy` workflow、部署门禁测试、`.gitignore`、工作区上传部署总览/总规则/历史核心记录。未修改 Cloud 业务代码、数据库 schema、真实 `.env`、服务器 `deploy-release.sh` 行为或 Edge/AICopilot 运行代码。
- 改动原因：旧日常路径依赖 self-hosted runner 构建镜像，多次出现等待、超时和误读旧文档问题；用户明确要求 Cloud/AICopilot 标准生产发布改为“先 push GitHub -> 本机构建镜像 -> 推 Harbor -> 本机 SSH 触发服务器 deploy-release.sh -> 服务器 pull/restart/health/cleanup”。
- 影响面：Cloud 日常发布入口改为本机 `deploy/scripts/local-release.sh --services <services>` 或显式 `--all`；`cloud-image` / `cloud-deploy` 只保留带确认词的灾备入口；服务器端 `deploy-release.sh` 继续作为唯一部署入口，负责 pre-check、backup、pull、restart、health、cleanup 和 history。
- 关键实现：新增 Cloud 本机 `build-and-push.sh`，支持显式 `--services httpapi,gateway,dataworker,migration,web`、`--all`、`--dry-run`、portable timeout、Harbor 登录检查、固定输出 `Release tag` / `Deploy services input` / `IIOT_*_IMAGE` 和 `artifacts/deploy/cloud-built-services.txt`；新增 `local-release.sh` 校验工作区干净、HEAD 已推送、SSH target 显式传入，并执行 `DEPLOY_GIT_SHA=<sha> DEPLOY_TRIGGERED_BY=local ./scripts/deploy-release.sh sha-<sha> --services <services>`；workflow push 日常触发被移除并要求 `EMERGENCY_CLOUD_IMAGE_BUILD` / `EMERGENCY_CLOUD_DEPLOY`。
- 规则提炼：形成长期部署规则，已写入工作区 `docs/总规则.md`、`docs/上传部署总览.md`、`docs/历史核心记录.md` 和 Cloud `docs/云端规则.md` / `deploy/README.md` / `deploy/OPERATIONS.md`：Cloud 生产发布不得等待 GitHub image/deploy workflow，本机构建、Harbor 检查和 SSH deploy 必须 fail-fast；超时后停止诊断 Docker buildx、Harbor tag、服务器 compose/logs 和 release 状态，不得继续 watch 或无限等待。
- 验证命令：
  - `bash -n deploy/scripts/build-and-push.sh deploy/scripts/local-release.sh deploy/scripts/deploy-release.sh`
  - `ruby -e 'require "yaml"; ARGV.each { |f| YAML.load_file(f); puts "YAML ok: #{f}" }' .github/workflows/cloud-image.yml .github/workflows/cloud-deploy.yml`
  - `./deploy/scripts/build-and-push.sh --dry-run --services httpapi,httpapi,web`
  - `if ./deploy/scripts/build-and-push.sh --dry-run --services nope; then exit 1; else echo 'Cloud invalid service rejected'; fi`
  - `./deploy/scripts/local-release.sh --dry-run --services web --ssh-target root@10.98.90.154`
  - `dotnet test src/tests/IIoT.EndToEndTests/IIoT.EndToEndTests.csproj --filter ConfigurationGuardTests --logger "console;verbosity=minimal"`
  - `rg` 检查旧标准入口描述和 fail-fast 规则。
- 验证结果：Shell 语法检查通过；Cloud workflow YAML 解析通过；dry-run 服务去重、非法服务拒绝、本机 SSH 命令打印通过；`ConfigurationGuardTests` 62 通过、0 失败；检索确认旧 `cloud-image` 日常标准入口描述已替换为灾备口径。
