# Cloud 前端架构整治计划

> 日期：2026-06-24
> 范围：仅 `IIoT.CloudPlatform/src/ui/iiot-web`
> 状态：已合并 CC 复核补充，可作为 Cloud 前端开工计划

## 1. 本轮边界

本计划只处理 Cloud 前端，不修改 `AICopilot`，不修改 `IIoT.EdgeClient`，不创建跨项目共享包。

当前 AI 端正在独立整改，AICopilot 的方案只作为参考。等 AI 端稳定且用户明确授权跨项目写入后，再评估是否抽取 `@iiot/ui-kit` 和 `@iiot/core`。

本轮明确不做：

- 不改 Cloud 后端 API 契约。
- 不改设备、权限、配方、过站、发布等业务规则。
- 不先切换到 pnpm workspace。
- 不先创建根目录 `packages/`。
- 不用一轮大重写替换全部页面。
- 不删除真实业务链路、真实错误反馈、真实权限校验。
- 不新增未批准依赖；CC 已建议批准 `vitest`、`@vue/test-utils`、`happy-dom` 用于 Cloud 前端测试，实际安装放到对应实施阶段。

## 2. 现状判断

Cloud 前端不是单点问题，而是缺少与后端 DDD 类似的前端边界。

当前已确认的问题：

- 多个页面 SFC 巨型化：
  - `EmployeeList.vue` 1277 行。
  - `ClientReleaseCenter.vue` 1271 行。
  - `RecipeList.vue` 1134 行。
  - `DeviceList.vue` 865 行。
  - `PassStationList.vue` 811 行。
  - `CapacityDetail.vue` 706 行。
- 页面承担了过多职责：模板、列定义、弹窗、抽屉、表单状态、CRUD、权限判断、错误处理混在一起。
- `UiDataTable` 当前以 render 函数为主要扩展方式，导致大量 `h()` 堆在页面内。
- `useListPage` 已存在，但列表页没有制度化使用。
- HTTP 客户端能解析 ProblemDetails，但错误路径仍分散在 axios 异常、`ApiResult<T>` 和页面级 catch 里。
- DTO 和 API 调用集中在 `src/api`，存在跨模块类型复用，例如 `device.ts` 从 `employee.ts` re-export `Pagination`。
- 前端没有单元测试脚本，`package.json` 当前只有 `dev`、`build`、`preview`。
- Cloud 使用 `npm + package-lock.json`，直接改 pnpm workspace 会扩大 CI/部署影响。

## 3. 目标架构

先在 Cloud 内部形成可迁移架构，未来再抽共享包。

目标目录：

```text
src/
  shared/
    ui/                 # 当前 Ui*、Niond*、状态组件的归口，后续可迁移到 @iiot/ui-kit
    feedback/           # toast、confirm、problem 展示
    table/              # DataTable 类型、formatter、通用 cell 组件
  core/
    http/               # axios client、ProblemDetails、ApiResult unwrap
    auth/               # auth store 支撑函数、refresh lock、storage sync
    permissions/        # 权限工具和类型
    types/              # Pagination、PagedList、PagedMetaData 等通用 API 类型
    list-page/          # useListPage 与分页模型
  features/
    devices/
      api.ts
      types.ts
      columns.ts
      routes.ts
      useDevices.ts
      DeviceListPage.vue
      DeviceRegisterModal.vue
      DeviceEditModal.vue
      DeviceDetailDrawer.vue
      DeviceDeleteConfirm.vue
    employees/
    recipes/
    pass-station/
    capacity/
    device-logs/
    client-releases/
    roles/
    master-data/
```

约束原则：

- 页面组件只做布局、子组件组合和命令连接。
- API 调用放 feature 内 `api.ts`。
- DTO 类型放 feature 内 `types.ts`，通用类型放 `core`。
- 表格列定义放 `columns.ts`，不继续堆在页面 SFC。
- 弹窗、抽屉、复杂表单必须拆独立 SFC。
- 列表页必须使用统一 list-page composable，避免每页手写 loading/page/error。
- 后端 ProblemDetails 的 `detail`、`errors`、`title` 不得被固定死文案覆盖。

### 阶段 1：Cloud 内基础设施收口

不换底层技术栈，不切 pnpm，不抽跨项目包。

优先做 Cloud 内部整理：

- 从 `utils/http.ts` 拆出：
  - `core/http/problemDetails.ts`
  - `core/http/apiResult.ts`
  - `core/http/httpClient.ts`
- 保留 axios，先统一错误解析和通知路径。
- 将 `types/api.ts` 的 `any` 改为 `unknown` 或明确类型。
- 将 `useListPage.ts` 移入 `core/list-page` 或保持现路径但补强能力。
- 梳理 auth store，可先只抽纯函数，不改行为。
- 统一 `Pagination`、`PagedList`、`PagedMetaData` 的来源，放入 `core/types/pagination.ts`，禁止 feature 之间互相 re-export 通用类型。

`core/types/pagination.ts` 目标类型：

```typescript
export interface Pagination {
  PageNumber: number;
  PageSize: number;
}

export interface PagedMetaData {
  totalCount: number;
  pageSize: number;
  currentPage: number;
  totalPages: number;
}

export interface PagedList<T> {
  items: T[];
  metaData: PagedMetaData;
}
```

```bash

rg "interface Pagination|interface PagedMetaData|interface PagedList" src/core/types
rg "ApiResult<T = any>|any\\[\\]" src/types src/core src/utils
```

### 阶段 3：按风险拆其他页面

按业务风险和复用价值排序，不全量并发：

1. `EmployeeList.vue`
   - 人员、账号、设备授权、角色绑定、个人特批权限。
   - 重点守住 `Employee.UpdateAccess` 与角色权限边界。
2. `RecipeList.vue`
   - 配方版本化，不得原地覆盖旧版本。
   - 参数编辑器必须拆独立组件。
3. `ClientReleaseCenter.vue`
   - 发布数据必须真实，不伪造安装包、hash、大小、更新说明。
   - 发布权限与设备权限分离。
4. `PassStationList.vue`
   - 必须从 `GET /api/v1/human/pass-stations/types` 读取 schema 动态渲染。
   - 不为普通工序新增独立 Vue 页面。
5. `CapacityDashboard.vue`、`CapacityDetail.vue`、`DeviceLogList.vue`
   - 保持真实归档和日志，不伪造统计、趋势、时间或告警等级。
6. `RoleList.vue`、`ProcessList.vue`
   - 体量较小，放在后面收口。

每个页面拆分都按同一套结构：

```text
features/{domain}/
  api.ts
  types.ts
  columns.ts
  use{Domain}.ts
  {Domain}Page.vue
  components...
```

每个切片完成后单独 build，不把多个高风险页面堆到一次提交里。

### 阶段 4：表格系统升级

当前 `UiDataTable` 依赖 render 函数，短期不能一刀切禁止 `h()`。

分两步做：

1. 短期：
   - 页面 SFC 不写复杂 `h()`。
   - 列定义移到 feature `columns.ts`。
   - 通用 cell 提取为小组件，例如状态 Tag、操作按钮组、mono text、release note cell。
2. 中期：
   - 增强 `UiDataTable`，支持 formatter、named slot、cell component。
   - 新增列配置优先使用声明式 formatter。
   - 逐步减少 render 函数。

验收：

```bash

```

### 阶段 6：API 类型生成评估

这一步放后面，不作为第一阶段前置。

原因：

- 当前更大的问题是页面结构和错误链路，不是单纯缺类型生成。
- 引入 `openapi-typescript` 是新增依赖，需要确认。
- 类型生成要看 Cloud HttpApi 的 swagger 输出是否稳定。

目标：

- 后端 OpenAPI 生成 TypeScript 类型。
- feature 的 `types.ts` 尽量 re-export 或组合生成类型。
- 手写 DTO 只保留前端视图模型，不冒充后端契约。
