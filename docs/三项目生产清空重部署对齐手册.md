# 三项目生产清空重部署对齐手册

本文档说明 CloudPlatform、AICopilot、EdgeClient 的从零部署契约。唯一执行入口是工作区 `deploy/Deploy-FromZero.ps1`；真实值在本机只存 macOS Keychain，仓库只保存无值 schema。

## 1. 总原则

- 本次按生产清空重部署执行，不做旧接口、旧 DTO、旧 hash、旧 token、旧配置路径或旧表运行时兼容。
- 当前生产仍按内网受控 HTTP-only 基线执行；HTTPS/TLS/443/HSTS/证书专项不混入本轮。
- 服务器清空前唯一需要保留的数据是 Cloud 客户端更新历史元数据；设备、产能、日志、生产数据、配方、人员业务数据、AI 会话、RAG、Qdrant、RabbitMQ 消息等不保留。
- 客户端更新历史只保留 Cloud 下载中心/客户端发布管理可见的版本记录、更新内容、状态、删除原因和失败原因；旧安装包二进制不在本轮保留范围内，除非发布前另行决定保留 `EDGE_UPDATES_DIR`。
- 所有根密钥必须在远端改动前预存 Keychain；数据库清空后失效的 Edge Release API key 等派生凭据可在对应阶段重新签发并覆盖写回。
- 从零部署不自动创建设备、注册 `ClientCode` 或轮换设备 bootstrap secret。

## 2. 密钥来源

- canonical 键名与目标变量来自工作区 `deploy/` 下不含值的 secret schema。
- 标准流程只读 macOS Keychain；缺键时一次性只报告键名，并在任何 SSH、清空、migration 或凭据签发前停止。
- 旧 `部署私密资料/三项目生产部署私密手册.local.md` 和 `~/.config` env 只作为一次性迁移来源。迁移不得输出值、不得自动删除旧文件；迁移完成后标准流程不回退读取。
- checkpoint、summary 和日志只记录键名、阶段、hash/状态，不记录 secret 值。

## 3. 清空前保留客户端更新历史

在旧 Cloud 仍能访问数据库时执行：

```bash
cd /data/iiot-platform/cloud/deploy
./scripts/export-client-release-history.sh
```

脚本只导出以下表：

- `edge_client_release_components`
- `edge_client_release_versions`
- `edge_client_release_artifacts`
- `edge_client_release_retention_policies`

导出产物在：

```text
deploy/backups/client-release-history/client-release-history-<timestamp>.sql
deploy/backups/client-release-history/client-release-history-<timestamp>.sql.sha256
```

必须把 `.sql` 和 `.sha256` 放到清空不会影响的受控 checkpoint/backup 根，并把路径和 checksum 写入本次 checkpoint，不写私密手册。

## 4. 清空后恢复客户端更新历史

Cloud 重新部署并完成 migration 后，在启动正式使用前导入：

```bash
cd /data/iiot-platform/cloud/deploy
./scripts/import-client-release-history.sh /path/to/client-release-history-<timestamp>.sql
```

默认要求目标 release 表为空。只有修复失败导入时才允许：

```bash
ALLOW_CLIENT_RELEASE_HISTORY_IMPORT_OVERWRITE=true ./scripts/import-client-release-history.sh /path/to/client-release-history-<timestamp>.sql
```

导入后验收：

- Cloud 管理端客户端发布页必须能在历史版本弹窗看到旧版本记录。
- `GET /api/v1/human/client-releases/catalog?channel=stable&targetRuntime=win-x64&includeArchived=true` 必须返回历史版本。
- Public download 和 Edge 更新 catalog 仍只返回可分发状态，不得把 `Deleted` 包重新变成可安装版本。

## 5. CloudPlatform 部署对齐

- 真实 `.env` 由从零入口按 Keychain schema 生成并设受限权限；`JWTSETTINGS__SECRET` 至少 32 字符。
- `JwtSettings.Validate()` 对已配置 secret 执行 32 字符运行时下限；生产缺失 secret 由 `JwtSecretResolver` fail-fast。
- `Bootstrap Auth RequireSecret=false` 路径不得恢复，bootstrap secret 恒必需。
- 当前 HMAC 对称 JWT 方案是已知中期风险，RS256/JWKS 改造单列后续安全专项，不能从计划中消失。
- 部署前运行 Cloud preflight、容器非 root readiness、配置 guard 测试和 release history 导入验收。

## 6. EdgeClient 对齐

- Launcher 密码只能通过 UI 修改链路写入，唯一 hash 入口是共享 `EdgePasswordHasher`。
- 旧 64 hex SHA256 只触发强制重置，不允许正常登录。
- 新密码至少 10 位，包含大小写字母、数字和符号。
- 本地 Launcher 连续失败 5 次后短时锁定，成功登录或改密成功清零。
- 仓库和 preflight 按内容扫描 PowerShell 密码上下文中的 SHA256/ComputeHash，不能靠脚本文件名门禁。
- Edge 发布仍走 Cloud 客户端发布链路，release notes 必填；历史记录导回后继续在 Cloud 发布管理中可见，且不得为重发删除。宿主发布恢复历史之后的新 patch；同版本插件只允许从冻结候选重建并按 hash/size 与下载链完成对账，字节不一致时停止。
- Cloud 数据清空后重新签发 Edge Release API key并覆盖写回 Keychain；不得注册任何现场设备或修改其 bootstrap secret。

## 7. AICopilot 对齐

- fresh DB seed 会创建私有 MiniMax OpenAI-compatible 模型，默认占位 URL、无 API key、禁用。
- 从零入口从 Keychain 生成生产 `.env` 的 `AICOPILOT_PRIVATE_MODEL_*`，migration worker 在新库播种真实私有模型；API key 入库前必须加密为 `encv2:`。
- 私有模型生产参数：
  - provider：`MiniMax Private`
  - model：`MiniMax-M3-AWQ-INT4`
  - context window：`65536`
  - max output：`4096`
  - protocol：`OpenAICompatible`
- 生产 `.env` 必须显式配置：
  ```dotenv
  AICOPILOT_PRIVATE_MODEL_ENABLED=true
  AICOPILOT_PRIVATE_MODEL_PROVIDER=MiniMax Private
  AICOPILOT_PRIVATE_MODEL_NAME=MiniMax-M3-AWQ-INT4
  AICOPILOT_PRIVATE_MODEL_BASE_URL=<private-model-base-url>/v1
  AICOPILOT_PRIVATE_MODEL_API_KEY=<private-model-api-key>
  AICOPILOT_PRIVATE_MODEL_CONTEXT_TOKENS=65536
  AICOPILOT_PRIVATE_MODEL_MAX_OUTPUT_TOKENS=4096
  AICOPILOT_PRIVATE_MODEL_TEMPERATURE=0.2
  ```
- 模型 smoke 使用 `AICOPILOT_MODEL_SMOKE_*` 独立验收；如果网关要求任意 key，也必须在真实 `.env` 显式填写并留痕。
- AI 仍只能只读 Cloud，不新增 Cloud 写能力、隐藏 adapter 或后台写任务。

## 8. 验收顺序

1. 旧 Cloud 导出客户端更新历史并保存 checksum。
2. 清空服务器业务数据和容器卷，保留导出的 release history 文件。
3. 部署 Cloud，运行 migration，导入客户端更新历史。
4. 部署 AICopilot，确认私有模型 seed、密钥加密、模型 smoke、Cloud OIDC。
5. 发布 EdgeClient 新宿主 patch；对已存在的同版本插件重建并按 hash/size 对账，确认新宿主版本、已恢复历史版本和插件下载链同时可见。
6. 重新签发 Edge Release API key并写回 Keychain，发布 Edge 产物，验证 catalog/hash/download。
7. 只运行配置、migration、seed、权限、健康、下载链、checkpoint 与恢复验收；不运行三仓全量、coverage、mutation、duplication 或三端质量对齐。

每一步保存 checkpoint，失败从最后一个已确认阶段恢复；不得重新清空或无条件全量重建。

## 9. CC 复验重点

- G1：Edge Launcher 本地登录强密码和锁定是否真实执行并有测试。
- G2：JWT 对称签名风险是否进入后续 RS256/JWKS 专项，未被沉默跳过。
- G3：Cloud `JwtSettings.Validate()` 是否补运行时 secret 长度下限。
- G4：新增契约测试是否有真实断言，不是空壳 `Assert.True(true)`。
- R1：删除密码 hash 脚本前是否确认 UI 改密链路完整，并用内容扫描防止脚本改名绕过。
