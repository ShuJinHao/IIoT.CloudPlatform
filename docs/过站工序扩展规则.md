# 过站工序扩展规则

本文档约束 `IIoT.CloudPlatform` 的过站数据扩展方式。当前云端已经统一为“公共检索列 + `payload_jsonb` 扩展字段 + 配置驱动 schema”的模型。

跨 Cloud 与 EdgeClient 的完整接入清单见 `../../docs/新工序接入手册.md`。

当前生产活动 schema 只允许 `cp / 正极模切` 与 `ap / 负极模切`。注塑、叠片、匀浆属于未投产或已退役工序，不得进入生产 schema、catalog、seed 或默认 UI；其上传必须按未知 `typeKey` 拒绝。

## 固定规则

- 新增普通过站工序时，默认只改 `src/hosts/IIoT.HttpApi/config/pass-station-types.json`。
- 不再为每个工序新增独立 DTO、Event、Mapper、SQL、Consumer、Vue 页面或 RabbitMQ endpoint。
- 统一上传接口为 `POST /api/v1/edge/pass-stations/{typeKey}/batch`。
- 统一查询接口为 `GET /api/v1/human/pass-stations/{typeKey}` 和 `GET /api/v1/human/pass-stations/{typeKey}/{id}`。
- AI 高频生产数据统一查询接口为 `GET /api/v1/ai/read/production-records`。新工序只要进入 `pass_station_records + pass-station-types.json`，默认具备 AI 通用读取能力，不再为 AI 按工序新增独立查询接口。
- 前端过站页面必须从云端 `GET /api/v1/human/pass-stations/types` 读取 schema 动态渲染。
- 工序扩展字段必须进入 `payload_jsonb`，公共检索锚点保留为普通列。
- 标准 `CellDataBase` 传输元数据可以随批次进入统一链路；业务扩展字段仍必须由当前 `typeKey` schema 明确声明。未知业务字段严格拒绝，Human/AI 只投影公共列与 schema 字段，不返回 raw `payload_jsonb`。

## 公共列

`pass_station_records` 固定保留以下普通列：

- `id`
- `type_key`
- `device_id`
- `barcode`
- `cell_result`
- `completed_time`
- `received_at`
- `deduplication_key`
- `payload_jsonb`

只有这些字段可以直接参与通用高频过滤和排序。新增工序字段如果需要高频过滤、排序或统计，必须单独评估是否提升为普通列，或新增 JSONB 表达式索引。

CP/AP 将弹夹号映射到 `barcode`，并声明 `plcCode`、`plcName`、`startTime`、`punchingQuantity`、`punchingSpeed`。开始和完成时间以 UTC 持久化；单台 PLC 高频查询由 `(type_key, payload_jsonb->>'plcCode'/'plcName', completed_time)` 表达式索引覆盖。

## 配置项

每个工序配置必须包含：

- `typeKey`：工序类型键，必须唯一，推荐与工序编码的小写形式一致。
- `displayName`：中文显示名。
- `description`：前端页面说明。
- `supportedModes`：支持的查询模式。
- `fields`：工序扩展字段定义。
- `listColumns`：列表展示列。
- `detailSections`：详情页分组展示字段。

字段支持类型：

- `string`
- `number`
- `integer`
- `boolean`
- `datetime`
- `enum`

字段支持规则：

- `required`
- `maxLength`
- `min`
- `max`
- `unit`
- `precision`
- `options`

云端启动时会校验配置。`typeKey` 重复、字段 key 重复、字段 key 不安全、列表或详情引用不存在字段，都会导致服务启动失败。

## 新增工序步骤

以中性占位工序 `<typeKey>` 为例：

1. 在 `src/hosts/IIoT.HttpApi/config/pass-station-types.json` 增加一个 `types` 配置项。
2. 设置唯一、安全且与正式工序编码一致的 `typeKey`。
3. 在 `fields` 中定义该工序真实可采集的专属字段。
4. 在 `listColumns` 中选择列表展示字段。
5. 在 `detailSections` 中安排详情页分组。
6. 运行后端构建和前端构建，确认 schema 校验、上传校验和页面渲染通过。

除非新增公共检索字段、全新业务链路或新的持久化策略，否则不得新增 per-process 代码文件。

生产启用必须由管理员显式创建工序主数据和对应中文客户端设备；部署、migration 与 seed 不得自动创建设备、注册 ClientCode 或轮换 bootstrap secret。正式退役工序必须同时清除活动 schema、示例和入口；必要历史发布审计只归档并从 catalog 隐藏。

## 允许新增代码的例外

只有以下情况可以突破“只改配置”的规则：

- 新字段需要成为系统级公共检索锚点。
- 需要按 JSON 内字段做高频过滤、排序或聚合统计。
- 需要全新的业务链路，不再属于普通过站追溯。
- 需要独立权限、审计或状态流转规则。

出现这些情况时，必须先说明影响范围，再单独开小 PR，不得混入普通工序扩展。

## 验证要求

新增或调整工序配置后，至少执行：

```powershell
pwsh scripts/tests/Select-CloudCiTests.ps1 -Mode Default -BaseRef <真实祖先 SHA> -HeadRef HEAD -OutputPath artifacts/ci-selection.json
pwsh scripts/tests/Invoke-CloudCiSelectedTests.ps1 -SelectionPath artifacts/ci-selection.json -Configuration Release
cd src/ui/iiot-web
npm run build
```

只运行 selector 命中的受影响业务、架构、安全或发布契约验证。完整 E2E、全仓 build、coverage 和跨仓对齐仅在用户明确授权对应模式时运行。
