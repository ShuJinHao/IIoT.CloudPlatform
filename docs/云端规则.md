# 云端规则

本文档约束 `IIoT.CloudPlatform`。总规则见 `../../docs/总规则.md`。

## 0. 改动收口门禁

- 修改 `IIoT.CloudPlatform` 代码前，必须先读工作区 `AGENTS.md`、`../../docs/总规则.md`、项目 `AGENTS.md`、本文档、相关专题契约、相关源码、相关测试和近期 git/GitHub 历史。
- 已验收功能默认冻结；不能因为局部重构、测试修复、接口调整或文档整理顺手改变已有业务链路、权限边界、部署口径或数据契约。
- 每次代码改动完成前，必须更新项目滚动复盘文档 `改动复盘与规则沉淀.md`，最新记录放在最前。
- 每次复盘必须写明改动范围、改动原因、影响面、验证命令、验证结果和规则提炼结论。
- 必须判断是否形成长期规则；有则写入本文档、专题契约或工作区长期规则，无则在复盘中写明“无新增长期规则”及原因。
- 最终回复必须列出复盘文档、规则沉淀位置和验证命令；缺任一项，不得称为完成。
- 默认只更新项目滚动复盘文档，不为每个任务新增单独流水文档；只有形成可长期复用的业务或技术契约，才新增专题文档。

## 1. 云端定位

云端是制造业务主数据和长期归档查询的主要来源，负责：

- 人员、账号、身份、权限。
- 设备注册、设备主数据、设备删除约束。
- 配方和配方版本。
- 客户端 bootstrap/寻址。
- 产能、日志、生产数据、过站数据的归档和查询。

## 1.1 云端部署传输红线

- Cloud 日常应用发布只能走工作区 `deploy/Deploy-Changed.ps1`：先要求本地 `main` 已提交并自动 push 到 `origin/main`，再以生产 SHA 到远端 tip 的改动和项目依赖闭包只构建、推送 Harbor、部署受影响镜像。影响无法安全归属必须停止，禁止退化成全量。`Deploy.ps1 -Target Cloud -Services ...` 只允许作为统一入口内部执行器或显式恢复入口。`Auto` 在 SSH TCP 不可达时可经 `cloud-routine-request.yml` 交给同一 stable Runner，不得改走旧 `cloud-deploy` 或直调项目脚本。
- Cloud 的部署策略架构测试 `deploy/tests/TestDeploymentPolicy.ps1` 必须接入 `dotnet build` 和 `cloud-ci`；删除生产状态 inspect、显式服务集合、增量红线或重新把全量作为日常默认时，构建必须失败。
- 稳定服务器 Runner 常规安装/升级只能通过 `Deploy.ps1 -InstallRunner`；SSH 不可达且 Runner 完全缺失时，`cloud-routine-request.yml` 只允许从已推送提交一次性安装 pinned Runner，已有文件不得覆盖。日常应用发布不得同步 Compose、nginx 或其它服务器 scripts；基础设施变更必须使用独立维护窗口和旧事务入口。
- Runner 必须以 `github-runner` 非 root 执行；root 仅允许一次性修复目录 owner/mode。日常更新只允许选中应用服务并使用 `--no-deps`，不得顺带重建 PostgreSQL、Redis 或 RabbitMQ。
- migration 发布必须先生成非空 PostgreSQL 备份和 checksum；运行服务健康失败必须恢复发布前应用镜像。健康发布后的 cleanup/Harbor GC 不得留在同步热路径。
- 当前生产部署基线是内网受控 HTTP，不启用 HTTPS/TLS/443/HSTS，不把证书采购、证书签发或 HTTPS 改造作为当前云端治理计划的阻断项。
- 未经用户在当前轮明确批准证书预算、证书来源和允许修改目录，Cloud 不得新增 nginx 443 listener、TLS 证书挂载、HSTS、HTTPS redirection、`UseHttpsRedirection` 或强制 HTTPS 配置。
- HTTP 基线只允许部署在受控内网、VPN 或等价专线环境，不得直接暴露公网；公网暴露、外部客户访问或跨不可信网络访问必须先重新评审传输安全方案。
- 因 HTTP 不提供传输加密，必须用补偿控制降低风险：强 secret、短期 JWT、bootstrap secret 恒必需、限流、最小权限、runner/SSH 权限收敛、日志脱敏、部署前 fail-fast 检查和网络边界控制。
- 文档、测试和部署脚本不得把 HTTP 基线误报为待修复失败项；安全计划应优先修默认凭据、弱 secret、bootstrap 绕过、身份权限、runner/SSH 和 AI 只读边界。
- 部署 README 中可复制的生产 HTTP OIDC 示例必须使用 `<cloud-private-ip>` / `<aicopilot-private-ip>` 这类私网 IPv4 占位，不得把 `.internal.example` 模板域名写成可上线示例。
- 运行时错误提示、UI placeholder、脚本默认值、示例 `.env` 和 workflow 默认值不得硬编码真实现场 IP；必须使用泛化 host 示例或要求操作者显式传入。测试用例和标明“现场记录，不是模板默认值”的历史校准记录可以保留真实 IP。
- 生产 HTTP 入口的响应压缩由 nginx 统一负责；HttpApi 不得同时启用 `ResponseCompression` 造成双重压缩。nginx gzip 只压缩文本、JSON、JS/CSS、XML、SVG 等类型，不得把 zip、nupkg、exe、octet-stream 等安装包或已压缩包类型加入 `gzip_types`。
- nginx 容器必须使用非特权内部端口承载 HTTP；外部端口可以继续由 compose 映射为 `GATEWAY_HTTP_PORT`，但 `iiot-web` 和 `nginx-gateway` 不得靠容器内 `80` 或 root 绑定低端口运行。
- 运维文档里的服务器本机 curl 示例必须从 `.env` / `GATEWAY_HTTP_PORT` 读取外部入口端口；不得写死 `http://127.0.0.1:80/...`，避免生产入口端口为 81 或其它值时误导验收。
- Cloud 生产发布前必须运行容器非 root readiness 探针，确认 OIDC PFX 可被 HttpApi 容器用户读取、Edge 更新根目录及既有 `installers` / `plugins` / `velopack` 子目录可被 HttpApi 容器用户写入、nginx 内部端口正向保持 `8080` 且未回退到 `80`；探针失败不得继续发布。

## 2. 人员和权限

- 人员入职必须创建账号，人员档案和身份账号是一体业务。
- 人员可先入职，后分配设备机台。
- 客户端修改类操作默认必须同时满足设备分配和功能权限。
- 新设备注册是管理员专属操作，不得放宽为普通业务权限。
- 默认使用角色模板承载常用岗位权限，不要求运维逐个账号手动维护权限点。内置模板包括 `DeviceAdmin`、`ClientInstallerOperator`、`ClientReleaseManager`、`ProductionViewer`、`RoleAdmin`、`HrAdmin`。
- `ProductionViewer` 可以读取设备、客户端发布和上位机/PLC 状态投影；Cloud 不提供 `EdgeHost.Manage`，不得在云端新增、修改、删除、启用或禁用现场 PLC。
- 员工绑定或变更角色属于访问权维护，必须具备 `Employee.UpdateAccess`；只改姓名、启停等基础档案不得隐式获得角色绑定能力。
- `Role.Read` 只允许读取角色名等绑定所需信息；定义角色和修改角色权限仍分别使用 `Role.Define`、`Role.Update`。
- 个人特批权限是例外机制，应优先用角色模板解决常规岗位权限。
- Human 登录和 Edge operator 登录必须执行 Identity lockout：连续失败计数、达到阈值锁定、成功登录清零失败计数；凭据校验阶段对账号不存在、账号停用、账号锁定、密码错误使用统一失败语义，不泄露账号是否存在或状态。
- 管理员重置密码不得隐式清除 lockout；只有显式设计清锁命令、写审计并经过规则评审后，才允许把密码重置和解锁合并。
- JWT 认证判断必须走 ASP.NET Core JwtBearer 验签管线；生产运行时代码必须保留 issuer、audience、lifetime、signing key 校验和零 `ClockSkew`，不得用 `ReadJwtToken` 读取未验签 claims 参与认证、授权、身份链或权限判断。
- `ReadJwtToken` 只允许测试解析由真实服务端签发的测试 token；Cloud 生产程序零例外，即使标称为日志、诊断或展示也不得绕过 JwtBearer 验签管线。`CLOUDARCH008` 以精确方法符号在编译期拒绝生产调用。
- `JwtSettings:Secret` 只要被配置就必须至少 32 字符；发布 preflight 和 `JwtSettings.Validate()` 都必须执行下限校验。生产缺失 secret 由 `JwtSecretResolver` fail-fast，开发环境自动生成临时 secret 不得被写回配置或文档。
- 当前 HMAC 对称 JWT 是已知中期安全风险：持有 Edge 配置 secret 的人理论上可伪造 Cloud token。该项必须作为后续 `RS256/JWKS` 令牌签名专项持续跟踪，不能在本轮 HTTP-only 或部署安全计划中沉默消失。
- Cloud Web 的 Vite/build/dev 配置不得枚举或输出 `process.env`，也不得打印原始 API target；按环境变量名做宽泛过滤不构成脱敏，例如 `PORT` 会误命中 `OTEL_EXPORTER_*` 并泄露 header。若需要启动诊断，只能输出固定、无值、无 secret 的稳定分类；ContractFilesystem required 测试必须阻断环境枚举和配置级 `console.log` 回归。
- 客户端发布权限必须和设备权限拆开：查看使用 `ClientRelease.Read`，生成客户端下载器使用 `ClientRelease.GenerateInstaller`，发布上传使用 `ClientRelease.Publish`，发布目录和保留策略管理使用 `ClientRelease.Manage`。
- Edge 日常发布不得依赖 Human refresh token 会话；Cloud 必须提供 Edge Release API key 换短期发布 JWT 的机器发布身份。机器发布身份只能携带 `ClientRelease.Read` / `ClientRelease.Publish`，不得获得 `ClientRelease.Manage`、设备权限或管理员绕过。
- Edge Release API key 明文只能创建时返回一次，数据库只保存高熵 key 的 hash；吊销、过期、最后使用时间和创建/吊销操作者必须可审计。

## 3. 设备和身份链

- 设备注册成功后由云端生成唯一 `ClientCode`。
- `ClientCode` 是客户端 bootstrap/寻址码，不是归档主键，不允许维护时手工改写。
- `DeviceId` 是云端正式设备标识，客户端上传产能、日志、生产数据必须使用 bootstrap 后获得的正式身份。
- 设备名称仅用于展示，不得作为上传身份。
- 权限判定、人员设备访问范围和设备身份必须每次从正式存储读取，不得复用故障可降级的普通值缓存。`DeviceIdentityQueryService` 的变更必须以真实 PostgreSQL + Dapper 行为测试证明 missing/current、底层事实更新后二次读取无 stale cache，以及调用方取消传播；取消反例必须使用唯一 `ApplicationName` 并有界观察 `pg_stat_activity + pg_locks` 已进入真实 Lock wait 后再取消，不得用固定延时猜测时序。
- 设备只允许修改名称。
- 设备删除是管理员高危级联硬删除，必须同时满足 `Device.Delete` 和 `Device.CascadeDelete` 权限点。删除设备时必须按 `DeviceId` 显式级联删除配方、产能、设备日志、过站数据、客户端版本快照、客户端运行心跳、上传幂等登记、人员设备授权、设备 refresh token 等关联数据。
- 设备删除影响查询和审计必须覆盖数据库会实际删除的关联状态，包括 `EdgeHostPlcRuntimeState`；不得让 FK cascade 或显式 SQL 删除静默发生而不在影响范围和审计摘要中体现。
- 设备删除前必须展示级联影响范围并二次确认；删除后必须写审计，记录操作者、设备、`ClientCode`、各类删除数量和结果。
- 人员设备授权必须直接读取正式存储；设备级联删除在同一事务删除对应授权行，不捕获员工 ID 或维护授权值缓存。
- 设备删除不得静默删除，不得只依赖数据库隐式 cascade。
- 设备级联删除写路径必须走 EF Core 基础设施服务，并和设备主数据删除、Outbox、审计、缓存失效保持同一业务用例编排；Dapper 只能保留归档读模型和历史记录写入，不得实现设备级联删除命令。对尚未建 EF 实体的历史表，允许在 EF 事务内使用参数化 SQL 删除。

## 4. 配方

- 配方归属于设备和工序。
- 初始版本从 `V1.0` 开始。
- 配方修改必须创建新版本并保留旧版本用于追溯，不得原地覆盖当前版本。

## 5. 过站工序扩展

- 云端普通过站工序采用“公共检索列 + `payload_jsonb` 扩展字段 + 配置驱动 schema”。
- 新增普通过站工序默认只改 `src/hosts/IIoT.HttpApi/config/pass-station-types.json`。
- 不为每个普通工序新增独立 DTO、Event、Mapper、SQL、Consumer、Vue 页面或 RabbitMQ endpoint。
- 统一上传接口：`POST /api/v1/edge/pass-stations/{typeKey}/batch`。
- 统一查询接口：`GET /api/v1/human/pass-stations/{typeKey}` 和 `GET /api/v1/human/pass-stations/{typeKey}/{id}`。
- AI 高频生产数据统一读取接口是 `GET /api/v1/ai/read/production-records`；新工序只要进入 `pass_station_records + pass-station-types.json`，默认具备 AI 通用读取能力，不再为 AI 按工序新增独立查询接口。
- 只有新增公共检索字段、高频 JSONB 查询/统计、全新业务链路、独立权限/审计/状态流转时，才允许突破配置扩展，并且必须单独说明影响范围。

## 5.1 客户端发布和设备状态

- 客户端安装状态只能来自设备版本上报和当前 release catalog 差异；客户端软件状态只能来自运行心跳，不得用版本上报超过 24 小时替代“未运行”判断。
- 运行心跳表必须按 `device_id + client_code` 唯一，保留 `last_heartbeat_at_utc` 索引用于超时查询；心跳上报必须校验 `DeviceId + ClientCode` 身份链。
- `DeviceClientState` 是客户端状态官方投影，按 `DeviceId + ClientCode` 唯一；版本上报和运行心跳进入 Cloud 后必须同步更新该投影。设备客户端状态页、AI Read 和后续状态类查询不得临时拼接 snapshot/heartbeat 重新解释状态。
- 客户端 `softwareStatus` 只能由共享纯判定器根据 `DeviceClientState?`、显式 `utcNow` 和 `LastRuntimeHeartbeatAtUtc` 生成；版本上报、版本号和投影 `UpdatedAtUtc` 不得刷新运行新鲜度。只有超过 24 小时才是 stale，恰好 24 小时仍是新鲜。
- Human 上位机与客户端版本清单必须复用同一 `softwareStatus` 判定器、每次 handler 只捕获一次 `utcNow`，并保持已验收的状态值和中文 Issue 文案。
- 客户端发布运行时聚合根是 `ClientReleaseComponent`；版本集合子实体是 `ClientReleaseVersion`，受控文件清单子实体是 `ClientReleaseArtifact`，二者不得作为独立聚合根或仓储入口；宿主和插件不得再维护两套 release 聚合、两套仓储或目录扫描补版本链路。
- 客户端发布追加新宿主/插件版本时，只允许插入新的 `ClientReleaseVersion` 和 `ClientReleaseArtifact`；不得顺手更新未变化的 `ClientReleaseComponent` 元数据或 `UpdatedAtUtc`。只有组件元数据实际变化、已有版本被覆盖更新、状态变更、归档或删除时，才允许触碰组件更新时间，避免生产导入历史版本后无意义更新旧聚合根导致发布上传失败。
- Human 客户端发布上传 Controller 只能发送保留 `ClientRelease.Publish` 权限元数据的无参数业务命令；应用命令和上传来源端口不得携带 `Stream`、`HttpContext`、Request/Body/ContentType/RemoteIp 等传输对象或命名。HttpApi 只在宿主适配器内读取当前请求，向应用层最多暴露声明长度、可取消的字节读取和可空审计来源字符串。
- 宿主 bundle 与独立插件包是两个真实发布用例，必须保留各自 manifest、hash、落盘、聚合、回滚和保留策略语义；但二者必须复用唯一上传协调器/会话，统一负责发布根解析、限速、最大体积、声明长度核对、暂存目录和清理。不得再复制上传循环、锁 helper、暂存 finally 或为了“统一”误删任一业务用例。
- Host bundle 与 Plugin package 的发布互斥只允许使用同一个稳定 Redis 资源键和 `5s` 获取总预算，并由 `DistributedLockBehavior` 覆盖完整 handler。不得在上传会话、协调器或 handler 内保留 `.edge-release-upload.lock`、`FileStream + DeleteOnClose`、本地锁 fallback、第二把锁或兼容开关。
- 客户端发布权限校验必须先于 Redis 租约获取、上传来源读取和暂存目录创建；未授权请求不得获取锁或读取任何上传字节。`OperationCanceledException` 必须在完成必要回滚和同步清理 staging 后继续向上传播，不得转成 `Result.Invalid`。提交点前取消必须回滚本次新建产物，提交点后取消不得删除已提交产物。只有本项目显式创建且文案已审查的验证异常可以把安全提示返回 HTTP/审计；IO、第三方 `InvalidDataException` 和未知异常只能返回稳定通用错误。结构化服务日志只记录 EventId、稳定操作/目标和异常类型，不得写原始 `ex.Message`、绝对路径、secret 或 stack trace；清理失败不得使用空 `catch` 静默吞掉。
- ClientRelease 数据库提交异常后的“已提交 / 冲突 / 未知”判定不得使用同一 scoped `DbContext` / `IRepository` 冒充独立确认。基础设施只读端口只保留 expected-set 批量合同，每次调用创建 fresh `IIoTDbContext`，以 `AsNoTracking + filtered Include + AsSingleQuery` 在一条 SQL、一个数据库快照中读取全部目标组件、版本和 artifacts；单 identity overload、逐版本循环查询、轮询、失败 context 复用和同 scope fallback 必须物理删除。Plugin 以 singleton expected set 调用同一端口；Host expected set 必须包含 Host 和本 bundle 新建的全部 Plugin version。
- `IAuditTrailService` 的 EF 持久化每次调用必须使用已配置 `DbContextOptions<IIoTDbContext>` 创建并释放独立的短生命周期 `IIoTDbContext`，只提交当次 `AuditTrailRecord`；不得复用请求 scoped 业务 context、通过 `ChangeTracker.Clear/Detach` 补丁止损，或在写审计时顺带提交/清空业务 pending changes。调用方 cancellation 必须继续抛出；非调用方异常可 best-effort 收口，但日志只允许稳定 EventId 和 `ErrorType`，不得附带原始 exception/message、目标路径或 secret。
- 独立插件发布从调用 `SaveChangesAsync` 前一刻起即进入提交结果可能未知区：之后任何异常或取消都不得删除插件包。fresh 数据库观测与本次静态包目录、唯一文件名、sha256、size、ownership 同时 exact 时，只允许当前 handler 恢复成功；数据库状态不一致固定 `409 client_release_publish_conflict`，未观测、超时、读失败或静态事实不 exact 固定 `503 client_release_commit_unknown`。普通后续重复发布仍为 `409`，不得把本规则扩展成通用 exact retry、目录复用、fingerprint 或幂等表。
- 独立插件文件发布必须先在目标同父级唯一私有目录写入固定 32-byte ASCII ownership marker 和唯一包文件，再用 `Directory.Move(private,target)` 原子占位；只有原子移动成功才拥有 target。提交前回滚只允许删除本次 owned 且目录/marker/包名/hash/size/无额外条目全部 exact 的目录；竞争 target、marker 被替换、包被修改或出现额外条目时必须 fail closed，绝不能递归删除不明目录。marker 清理必须状态化幂等。Plugin 路径子集、SHA-256 和 regular-file exact 事实必须由唯一无状态 helper 提供，handler、文件事务和测试不得各自重写算法。
- 独立插件保存调用抛出 `OperationCanceledException` 时仍必须用与调用方 token 无关的独立 `5s` 观测完成一次事实确认，然后原样重抛取消；exact commit 写成功审计“已提交但响应取消/租约丢失”，mismatch/unknown 也必须使用 `CancellationToken.None` 写固定脱敏审计。审计和 HTTP 不得包含原始数据库错误、绝对路径、secret 或 stack。
- 插件 manifest 的 `createdAtUtc` 可为空；非空时只接受 `Z/UTC` 或明确 offset，无 offset 的 `Unspecified` 必须在保存前以 `400` 拒绝，禁止按服务器本地时区猜测。offset 只按其 `Kind` 转为 UTC 后持久化；fresh matcher 对 UTC 时间按 PostgreSQL `timestamptz` 的 `1µs` 精度严格比较，不得放宽超过一个微秒。
- Host bundle 从调用 `SaveChangesAsync` 前一刻起进入提交结果可能未知区；之后任何异常或取消都不得删除 installer、Velopack 或同包新生成 Plugin package。保存非取消异常只允许一次独立 `5s` expected-set 批量观测：全部数据库版本与本次唯一 expected matcher exact，并且 installer 目录、每个新 Plugin package、全部 Velopack 发布文件静态事实 exact，才可恢复成功；任一已观测数据库版本 mismatch 固定 `409 client_release_publish_conflict`，缺失/部分观测、超时、reader failure 或静态 mismatch 固定 `503 client_release_commit_unknown`。取消必须完成同一次事实观测和固定脱敏审计后原样重抛。
- Host manifest 必须在任何 Velopack、installer 或 generated Plugin 写入前完成模块 ownership 预检：`modules` 必须非 null、非空且不得含 null 项；trim 后的 `ModuleId` 与 slash-normalized `PluginDirectory` 分别按大小写不敏感保持唯一；每个 module version 必须是 SemVer，禁止把 raw traversal/非版本字符串用作发布目录段。任一失败固定按无落盘的非法 manifest 拒绝，不得依赖后续字典覆盖或回滚补救。
- Host installer 发布必须先在 upload staging 的 source 内验证无保留 marker 并写入 32-byte ASCII ownership marker，marker 成功后才允许移动到 target 同父级私有目录并以 `Directory.Move(private,target)` 原子占位；marker 创建/flush 失败时文件必须仍完全留在 upload staging，不能产生无主 private 目录。提交前回滚只允许删除目录、marker、相对路径、hash、size 和零 extra 全部 exact 的本次 owned tree，禁止递归删除不明目录。
- Velopack `.nupkg` 是不可覆盖发布文件：目标不存在时用同目录私有临时文件原子占位；目标已存在且 hash/size exact 时只复用且不取得回滚所有权，目标不同固定冲突。可变 manifests 必须先精确备份既有 regular file，再经同目录私有文件原子替换；提交前恢复必须验证当前发布事实和 backup hash/size，篡改、backup 缺失或 reparse ancestor 均 fail closed。发布目录创建必须逐级校验从受控 edge root 到目标/备份目录的所有既有 ancestor，禁止通过 symlink/reparse point 穿透。
- Installer、Plugin、Velopack 的 source、private、target、backup 目录创建、静态观测和回滚必须复用唯一受控目录链校验器；字符串上属于 edge root 子路径不等于物理受控，edge root 及路径上每个既有 ancestor 都必须是非 reparse 的真实目录。不得在各文件事务前再复制 lexical child 检查。目录快照必须逐目录使用 top-level 枚举，先拒绝 symlink/reparse child 再决定是否进入，禁止用未经证明不会跟随链接的递归枚举建立 ownership 事实。
- Host 顶层 `DownloadUrl/Sha256/PackageSize` 继续分别表示 installer manifest URL 与 installer stub 的 hash/size，不得借治理改变既有 catalog 语义；`ManifestFile`、installer stub `PackageFile` 和每个 `VelopackFile` artifact 必须记录各自真实 regular-file hash/size，不得用 installer stub hash 冒充 manifest 或给 Velopack 留空事实。`VelopackFile` artifact builder 的 sha256/size 必须是必填非空参数，不得保留允许生成不完整事实的 optional/default 兼容形态。
- Host/Plugin 只共享 expected-state 数据模型、唯一 exact matcher、fresh batch reader 和无状态 file-facts；两条 handler 的发布编排、文件事务、恢复结果和保留策略必须保持分离，不得抽取掩盖 installer/Velopack/Plugin 差异的通用 recovery workflow/base，也不得保留旧新双轨 helper。
- ClientRelease 中结果和副作用完全相同的机械能力必须保持唯一实现：wrapper ZIP 解包/entry 规范化/zip-slip 与 reparse 拒绝、SHA-256 格式与发布磁盘阈值、edge-updates 根解析、URL/路径段编码、channel/可空文本规范化、审计 actor 解析、归档 Plugin 目录删除，以及 Velopack manifest 名称与引用扫描不得在 Host/Plugin/查询/删除链各自复制。共享 helper 必须无业务编排或以显式参数/命名保留策略差异；Host/Plugin 的 manifest、artifact、retention、rollback、download、commit recovery 和审计结果仍必须分离，禁止借“去重”抽取万能 handler/base。
- 常规 Cloud CI 必须运行 ProductionService 全量测试和 `PostgresPersistenceIntegration` 真实 PostgreSQL 门禁。门禁同时覆盖 ClientRelease post-commit exception simulation 的 fresh recovery，以及 `DeviceIdentityQueryService` 对 missing/current、底层事实更新后二次直接读取和调用方取消的真实 Dapper 行为；Docker/PostgreSQL 不可用必须失败，不得 Skip 或用 SQLite/InMemory/源码字符串断言冒充。仓内 EndToEnd（不含 LiveExternal Cloud/AICopilot workspace alignment）仍保留手动重型入口；LiveExternal 对齐必须用独立显式环境与证据验收。
- 客户端宿主 bundle 上传和独立插件上传完成落盘后，服务端必须把受控发布目录、版本目录和文件权限收敛为 nginx gateway 可读可遍历、但不可 world-writable；不得依赖人工 `chmod` 修复静态下载 404。Linux/macOS 基线为目录 `755`、文件 `644` 或等价权限。
- Cloud 暴露 Velopack 更新源时，`/edge-updates/velopack/{channel}/` 下必须存在标准 `RELEASES` 文件；上传包如果只提供 `RELEASES-{channel}`，Cloud 发布端必须归一复制为 `RELEASES`，不能让 launcher 更新源依赖人工重命名。
- 客户端宿主 bundle 和独立插件包写入 `Published` 前，服务端必须再次校验受控静态文件链路可分发：installer artifact、安装器、Velopack `RELEASES`、channel manifests、当前版本 `.nupkg`、插件 zip 均必须存在、非空、位于 `edge-updates` 受控目录、无符号链接/重解析点且 nginx/static 可读；当前版本 `.nupkg` 还必须被 Velopack manifests 引用。失败时必须拒绝写入 `Published`，不能先发布再人工补文件或 chmod。
- 旧 `edge_client_host_releases` / `edge_client_plugin_releases` 只允许出现在历史 migration 和一次性迁移脚本中；运行时查询、发布、首装生成、状态变更、删除和保留策略不得再依赖旧表。
- 管理员物理删除发布包文件必须具备 `ClientRelease.Manage`，并先判断是否在用：只要 `edge_device_client_version_snapshots` 中有设备当前版本等于目标版本，就视为在用，不额外加时间窗口；在用包只能归档或改状态，不得物理删除。
- 发布 catalog、首装生成、Edge 更新 catalog 和公开下载 catalog 的版本来源只能是 Cloud 发布记录；不得在查询时扫描文件目录补出未登记版本，也不得让残留文件把 `Deleted` 包重新显示出来。
- 发布包物理删除只能由服务端 release 元数据和 `EdgeInstallerArtifactOptions.RootPath` 推导受控发布目录下的真实文件路径，不接受前端传入路径；删除前必须限制在受控发布根内，拒绝 `..`、符号链接/重解析点逃逸，并把删除文件清单写入审计日志。
- `Archived` 表示业务归档，文件可能仍存在；`Deleted` 表示已从可分发 catalog 和首装链路移除，发布历史、版本号、更新时间、更新内容、删除原因和失败原因必须保留。
- 生产服务器清空重部署时，默认不保留设备、人员、配方、产能、日志、生产数据、AI 会话、RAG、Qdrant 或 RabbitMQ 消息；唯一默认保留数据是客户端更新历史元数据。清空前必须导出 `edge_client_release_components`、`edge_client_release_versions`、`edge_client_release_artifacts` 和 `edge_client_release_retention_policies`，清空后只导回这些 release history 表，确保 Cloud 下载中心和客户端发布管理仍能查看历史版本记录。

## 5.2 上位机和 PLC 状态

- Cloud 上的“上位机”不是手工创建出来的配置聚合根，而是已注册 `DeviceId + ClientCode` 设备身份、Edge 客户端运行状态和客户端上报 PLC 清单/状态的只读呈现。
- Cloud 不得提供新增、修改、删除、启用或禁用上位机/PLC 的 Human、AI、public 或内部写入口；`EdgeHost.Manage` 权限点不得恢复。
- 上位机列表和详情必须以当前人员可访问的 `Device` 为主数据源，左连 `DeviceClientState` 和 `EdgeHostPlcRuntimeState`；不得以旧 `EdgeHost` 配置表作为列表基准。
- 上位机列表的 count、分页和 keyword 过滤必须在数据库侧完成；只允许为当前页设备批量读取客户端状态和 PLC 状态，不得把全部授权设备与全部 PLC 状态拉入内存后再分页。
- `edge_host_plc_runtime_states` 是 PLC runtime state 官方投影表，按 `DeviceId + ClientCode + PlcCode` 唯一；它不是聚合根，不得通过通用仓储写入，只能通过专用 store 维护。
- PLC runtime state 上报只能走 Edge 设备身份入口 `POST /api/v1/edge/edge-hosts/plc-runtime-states`，必须校验 `DeviceId + ClientCode` 身份链；不得要求 Cloud 侧存在 `EdgeHost` 配置记录。
- runtime state 不得依赖 `EdgeHostId` 或 `PlcBindingId`；迁移和后续模型维护必须以 `DeviceId + ClientCode + PlcCode` 为唯一身份清理重复数据。
- EdgeClient 的 `PlcCode` 是持久化且改名不变的 PLC 稳定身份，`ReportedPlcName` 仅用于展示；Cloud 不得用展示名称替代或重建 PLC 身份。
- 每次 PLC runtime state 上报都是当前 `DeviceId + ClientCode` 的完整快照，不是增量事件；本次缺失的旧编码必须在同一保存单元中删除，合法空列表必须清空该身份下的旧投影，无效请求不得部分修改投影。
- Human 端读取 PLC runtime state 只能走 `GET /api/v1/human/edge-hosts/{deviceId}/plc-runtime-states`，并使用 `EdgeHost.Read` 权限；该查询只能展示设备身份、客户端运行状态和状态投影，不得反向修配置。
- 同一次 Edge 上报不得包含重复 PLC 编码。
- EdgeClient 必须明确上报 PLC `RuntimeStatus`：已连接为 `Connected`；未连接且明确故障或存在 `LastError` 为 `Faulted`；尚无快照、未知或无错误连接中为 `Unknown`；其余已确认未连接状态为 `Disconnected`。Cloud 只对旧客户端做缺省兜底，未传状态且 `IsConnected=false`、`LastError` 非空时按 `Faulted` 展示。
- `edge_hosts` 和 `edge_host_plc_bindings` 是废弃的云端手工配置表；运行时查询、写入、播种、权限、审计和设备删除影响不得再依赖它们。
- 设备删除时必须显式清理该设备的 `edge_host_plc_runtime_states`，并在影响查询和审计摘要中体现删除数量。

## 6. 架构和代码

- 遵守现有 DDD 分层和依赖倒置。
- 修复 NuGet 版本冲突或 MSB3277 warning 时，必须先追溯依赖图和历史复盘；不得为了消除 warning 回退到已知漏洞版本或历史已验证不兼容版本。`Microsoft.OpenApi` 在 Cloud HttpApi 中不得降级到 `2.3.0`，也不得在未验证 `Microsoft.AspNetCore.OpenApi` source generator 兼容前升到 3.x；应优先对齐上游依赖包族，例如 Swashbuckle。
- `BaseEntity<TId>` 只表示普通实体和领域事件容器，不代表聚合根；真实聚合根必须显式实现 `IAggregateRoot<TId>`。
- 通用 `IRepository<T>` / `IReadRepository<T>` 只能用于显式聚合根。子实体不得实现 `IAggregateRoot`，也不得作为应用服务注入仓储的入口。
- 聚合内子集合必须由聚合根方法维护，对外只能暴露只读集合；不得在应用服务中直接新增、删除或更新子实体来绕过聚合不变量。
- 投影表、状态表、幂等登记、审计、Outbox、refresh token 等基础设施持久化模型不得为了复用仓储而伪装成业务聚合根。
- 状态/投影模型如需复用查询条件，应收在专用 store、读服务或基础设施实现内部，不在 Core 暴露面向通用仓储的 Specification 入口。
- Outbox 每次传输必须使用持久化 `OutboxMessage.Id` 作为稳定 transport `MessageId`；`IEventPublisher` 只保留这一个有真实调用方的显式 ID 合同。不得在重试时重新生成 ID，也不得保留无调用方的 generic overload、adapter、wrapper 或 fallback。
- DataWorker 的三个真实 MassTransit receiver 端点必须统一启用 PostgreSQL EF inbox/outbox，以 `MessageId + ConsumerId` 唯一约束实现消费幂等；重复检测窗口固定 7 天、query delay 固定 1 秒，由正式 EF migration 和清理配置同时约束。发送端 `Publish -> Save/Commit` transient 采用 at-least-once transport：事务回滚后必须以同一稳定 ID 重发；接收端 inbox 将它收敛为 exactly-once business side effect。ACK-lost 验收必须在真实 PostgreSQL 与真实 receiver 上证明“同一 ID 传输两次、业务副作用一次”，并检查 inbox consumed/delivered 与 cleanup 语义；不得用 fake consumer、固定延时或只检查发送端代替。
- `DataWorker:Testing:DisableOutboxDispatcher` 只允许精确 `Testing` 环境下的 receiver-only 集成测试使用；其它环境一旦配置为 true 必须在宿主构造时失败，生产 compose 必须显式固定 `DOTNET_ENVIRONMENT: Production` 且不得出现该开关。TestKit 只能设置 AppHost 的真实环境；AppHost 通过 `builder.Environment.EnvironmentName` 向 DataWorker 传递已验证环境，不得在子进程配置中硬编码 `Testing`。测试关闭 outbound dispatcher 不得关闭三个真实 receiver。
- 领域规则放领域层，应用编排放服务/用例层，基础设施细节放基础设施层，接口层只收发请求和结果。
- 不把业务判断散落到 Controller、缓存适配层或临时工具类。
- 数据变更影响缓存读取时，必须同步处理缓存失效、刷新或重建。Domain Event 严格失效和普通 `RemoveAsync` 都必须在物理 remove 前推进 exact-key generation，普通/严格 pattern 失效都必须在 endpoint scan 前推进全局 pattern generation；`GetOrSetAsync` factory 在 Fusion 写回前二次核对并 fail-closed 禁止旧 L1/L2/backplane 写入，正常 provider 完成后以独立有界 cleanup token 覆盖最后的 check→write 窗口。caller cancellation、ignored token 和 synthetic timeout 都不能留下旧 factory 回填；原请求可观察已开始时的值，后续读取必须 miss/fresh。
- Redis 分布式锁的唯一应用合同是可观测租约 `IDistributedLockLease`；租约必须暴露一次性 `OwnershipLost` 取消信号。不得保留只有 `IAsyncDisposable` 的旧合同、兼容重载或调用方无法感知续租失败的双轨实现。
- 锁获取必须使用高熵 owner token 和 Redis `SET NX + TTL`，续租与释放必须使用 compare-renew / compare-delete Lua 原子校验同一 token。续租返回 0、超时或异常必须立即、幂等地标记所有权丢失并结束续租循环；MediatR 处理器即使忽略取消也不得在租约丢失后返回成功。
- Redis 获取、续租、解锁以及续租任务收口都必须有显式可配置硬超时并在启动时验证；必须满足 `renewal cadence + 2 * operation timeout < lease TTL`。`acquireTimeout` 是获取与重试的单调总预算，每次 Redis `WaitAsync` 只能使用“剩余总预算与 operation timeout 的较小值”；零只允许一次不等待尝试。总预算截止是竞争 `409`，独立 Redis operation timeout 是不可用 `503`。获取超时或取消后底层 Redis 命令晚到成功时，必须用原 owner token 异步 compare-delete，不得留下无主租约。
- 锁竞争/获取等待截止对 HTTP 稳定映射为 `409`；Redis 不可用、操作超时和租约所有权丢失映射为 `503`。响应与日志不得包含锁 key、Redis 原始 message、绝对路径或 stack trace；锁保护的写链路不得降级为本地锁、无锁或继续写入。
- 不为未来可能场景提前加入多余抽象、迁移层、兼容层、老接口保留层。
- 开发阶段不保留旧 API、旧 query 参数、旧 DTO 字段、旧路径或双轨兼容；跨项目契约调整必须一次性对齐调用方和被调用方。

### 6.2 普通值缓存韧性

- 权威规则：[`CLOUD-CACHE-001`](contracts/caching-resilience.md)。
- 当前 `ICacheService` 只保留 `GetOrSetAsync`、`RemoveAsync` 和 `RemoveByPatternAsync`。10 个生产 caller 必须逐个显式传入无默认值的 `shouldCache` 与正值 TTL：权限/工序/设备/设备配方列表/日报分页允许合法空结果；配方详情与单日 summary 的 null 不缓存；两个 range 的空列表不缓存。产能 4 个 caller 可在同一 `ProductionService/Queries/Capacities` bounded context 内共用一个有4个真实调用方的 admission/key/TTL 策略实现，但 Human 授权顺序和每个 handler 的 token/空值语义必须由 caller matrix 直接锁定；该共享实现不得变成兼容 adapter/wrapper。已经没有真实调用方的分离 `GetAsync`/`SetAsync`、默认 TTL overload、adapter、wrapper 或 fallback 必须物理删除。
- `RedisCacheService` 的 `IOptions<DomainEventCacheInvalidationOptions>` wrapper 及其 `Value` 都必须在构造时 exact non-null、fail closed；不得把 null 延迟成运行时 `NullReferenceException`，也不得提供默认 options fallback。
- `GetAllDevices` admin、`GetRecipeById` device scope 和 `GetDailyCapacityPaged` no-access/specific-device 的授权顺序不得因缓存迁移改变；cache hit 不能绕过配方 device scope。caller token 必须进入 repository/query factory；底层合同不接 token 时必须在调用前和循环中显式检查。
- 普通 `RemoveAsync` 必须在 remove 前推进 exact-key generation，普通 `RemoveByPatternAsync` 必须在 endpoint scan 前推进全局 pattern generation，防止 Process/Permission/Capacity 的失效返回后被旧 factory 回填。generation bump 是失效成功语义的必要条件；Redis 连接/超时、取消和未知异常都必须在物理 remove/endpoint scan 前传播，不得在 bump 未生效时返回失效成功。只有 bump 成功后的 Fusion 物理删除或 endpoint/scan 基础设施白名单异常可按普通值边界记录脱敏降级。
- 设备/配方的数据变更缓存失效只走 Domain Event handler，以稳定 `OutboxMessage.Id + handler scope` 生成 Redis receipt；命令内直接删除、旧 alias、adapter、wrapper、fallback 和双写路径必须物理删除。正常成功与事务重试必须只产生一次逻辑失效；claim/renew/complete 的 Redis 等待必须绑定原 caller token，取消不得被迟到的 completed receipt 或 complete 成功吞掉。失败时使用独立有界 cleanup token best-effort compare-delete 释放 claim，释放超时记录 `ReleaseCleanupTimeout`，其它释放失败只记录稳定降级日志，不得遮蔽原业务异常或 caller cancellation/token；崩溃时由 lease 过期允许补偿重放，不得宣称进程崩溃下 exactly-once。
- `__iiot_system:` 是 Redis 系统 namespace；业务 key 删除、pattern/glob 删除和扫描结果都不得命中该 namespace。严格失效中 Redis 断连、取消、续租/完成 owner 丢失与未知异常必须传播，不得套用普通值缓存的受控降级。

## 7. AI 读取边界

- `IIoT.CloudPlatform` 是八个 `/api/v1/ai/read/*` 端点及其权限、过滤、DTO、限流和审计语义的 canonical provider owner；`AICopilot` 只拥有 typed consumer 和独立 live-test，不得重新定义 provider 语义。
- AICopilot 可按批准范围只读云端数据、规则和接口说明。
- 当前没有默认 Cloud 写 API 授权给 AICopilot。
- AI-facing 读取接口只使用 Cloud 正式业务身份，不接受调用方把展示名、设备码或临时参数伪装成正式主键。
- AI Read 设备主数据必须提供 `deviceId`、正式 `deviceCode`、`processId` 精确过滤，并与 keyword、delegated device scope、count 和分页在同一数据库查询中按 AND 相交；工序主数据必须支持 `/processes?processId=` 精确解析。已知但不支持的 `status`、`lineName`、`processName`、`updatedAt` 等设备过滤参数必须返回 400，不能静默忽略后返回未过滤集合。
- AI Read `device-client-states` 只允许 `AiRead.DeviceClientState`，必须以 delegated scope 内当前 `Device` 为主集，复用设备主数据查询在数据库侧完成 `deviceId/deviceCode/processId/keyword/maxRows` AND 过滤、count、稳定排序和分页。无投影的授权设备仍返回 Missing 行，投影专属字段和 `updatedAtUtc` 为 null；AI DTO 不返回 Human `issue`。已知未支持状态/更新时间参数必须返回 400。
- AI 对已开放数据域的正式读取必须通过 Cloud `/api/v1/ai/read/*` 只读 API，不让 AI 端用 Direct DB、Text-to-SQL、MCP、Agent 或隐藏适配器绕开正式端点；已开放的数据域包括设备、工序、客户端发布版本、设备客户端状态、产能汇总、小时产能、设备日志和通用生产数据。
- AI 通用生产数据只使用 `/api/v1/ai/read/production-records`，返回公共列和 schema 化字段；不得给 AI 返回 raw `payload_jsonb` 字符串，也不得继续暴露 AI 专用 `/pass-stations/{typeKey}` 语义入口。
- `DeviceId` 是 AI 读取产能、日志和过站记录的正式设备参数；`ClientCode`/`deviceCode` 只可用于设备查询或 bootstrap 寻址，不能冒充 `deviceId`。
- 当前 AICopilot 规则禁止读取未批准的配方主数据和配方版本；Cloud 不应为 AI 侧保留未被当前目标使用的配方读取兼容入口。
- Text-to-SQL 只作为低频探索兜底，不能替代已沉淀的日志、产能和生产数据 AI Read API；Text-to-SQL 必须只读、限行、限时、审计并限制表范围。
- AiRead `QueryScope` 除 GUID、时间、数字、布尔和严格闭集枚举外，所有请求字符串只能记录 `present`；返回范围摘要与审计摘要使用同一口径。AiRead 失败审计只记录稳定 ResultStatus 或异常类型，不得保存 `Result.Errors`、异常消息、stack trace、prompt、token、日志正文或其它自由文本原文。
- Cloud/AICopilot 契约改动必须增加真实 provider ↔ typed consumer 联合验收：隔离 Cloud Aspire 环境通过真实 Gateway HTTP 驱动独立 AI live-test，token 只经子进程环境传递。WorkspaceAlignment 执行前必须用 `scripts/tests/Get-CloudAiWorkspaceEvidence.ps1` 为当前 Cloud worktree 和显式 `AICOPILOT_REPOSITORY_ROOT`（或当前 Cloud 根的严格同级 AI 根）生成唯一 evidence marker，成对记录两仓精确 git root、完整 HEAD SHA 和稳定排序的候选源码 SHA-256 digest；测试内必须反向验证执行程序集归属的 Cloud 根与 AI 根。任一侧源码、root 或 HEAD 变化后旧结果失效；单仓绿测、StubHandler 或手写 JSON 不等于整体对齐。
- 若未来提供新的 AI-facing API，必须先明确契约、权限、审计、幂等、允许动作和禁止动作。

Aspire 端到端测试启动必须有明确超时和错误信息，不得在 Postgres、RabbitMQ、HttpApi、Gateway 或 DataWorker 未就绪时无限等待；ResourceNotifications 报告资源 healthy 后，测试 fixture 仍必须等待 Gateway `/internal/healthz` 真实返回 200，避免 HttpApi bus、Redis 或下游健康检查刚启动时首个请求偶发 503。当端到端 case 会消耗全局限流桶、停用种子管理员或修改共享数据时，不得在没有可验证重置/分组契约前改为全套共享 fixture；当前 63 case 保持独立 Aspire 环境。本地冷启动实测 43 分 13.8 秒，因此只将 nightly/manual `full-end-to-end` 调整为 backend step 60 分钟、job 90 分钟；push/PR required job 仍为 25 分钟，不得因此放宽。本机验证若只需冒烟，应优先跑 `CloudProductionFlowTests.InternalHealthz_ShouldReturnOk` 或对应窄过滤。

Cloud 日常生产应用发布标准路径是工作区 `../../deploy/Deploy-Changed.ps1 -Targets Cloud` 自动 push 已提交的 main、读取生产 SHA、按影响选择服务，再由 `Deploy.ps1` 从 fresh 远端 tip 隔离构建、推送 Harbor并请求稳定 Runner；`cloud-image` / `cloud-deploy` 只允许作为带确认词的灾备入口。
`deploy/scripts/build-and-push.sh` 是日常镜像实现；`local-release.sh` / `deploy-release.sh` 和工作区 `Invoke-WorkspaceDeploy.ps1` 只用于 Compose/nginx/scripts 等部署基础设施维护与旧事务恢复，部署口径不得分裂。
日常应用发布禁止同步 support；基础设施维护才必须同步并校验 compose、nginx 和服务器 scripts 白名单并显式排除 `.env`、证书、release state 与备份。旧 release/cleanup 锁规则只约束该维护事务。
日常入口必须实时输出稳定阶段。dry-run 和语法检查不能替代真实发布验收；构建后远端失败优先按 invocation 续传，cleanup/GC/深度检查不在热路径。
- 本地 timeout watchdog 必须同时终止自己的计时 sleep 和目标命令的受监督本地进程树，不得留下继承 stdout/stderr 的孤儿；部署脚本不得主动创建新会话逃逸监督。普通 build、SSH 或远端脚本非零必须保留真实退出码，只有达到时间上限才允许返回 124。
- support sync 必须与正式 release 共用全局互斥锁，并把同步后 manifest SHA-256 绑定到本次远端发布；另一发布改写 support 后必须在 rollout 前失败。HUP/INT/TERM handler 必须释放自有锁后按 129/130/143 显式退出。
- Cloud 日常发布必须 fetch 配置远端 tip，再从该 SHA 的临时 detached worktree 构建；不要求本地工作树 clean，也不得读取未推送修改。每次发布使用独立 invocation/request/build 目录，禁止 dry-run 或并行发布覆盖控制文件。
- Cloud support 变更前必须先持久化绑定 invocation/SHA/plan 的原子 transaction marker；入口必须在获锁前扫描 marker、lock、staging 和 recovery。SIGKILL、OOM、重启或子 rollout 非零后，没有 current state + marker 晋级证据就必须 durable blocked；有证据则绝不恢复旧 support，下一轮只清理旧 backup/staging/lock/marker。本地 precheck 对 stale/malformed lock 只读返回 75，不得在 transaction scan 前删除证据；阻断证据必须记录真实旧 invocation 及 marker/backup/staging/lock 路径。
- installer 和 release child 必须使用 `env -i` 白名单环境并运行在独立 session/process group；父进程判断子任务时必须直接检查 PGID，不得以组长 PID 存活代替整组存活。即使组长已退出，仍必须对存活子孙执行有界 TERM/KILL，确认整个进程组消失后才能恢复或释锁。dotenv 非法行只能报 file/line/固定 category，不得打印 key、value 或 token。
- 操作员 `.env` 与 release-owned 应用镜像状态必须分离。deploy/rollback 不得替换 `.env`；应用镜像经 `staged-images.env` 原子晋升到 `current-images.env`，compose 在 `.env` 之后加载该 release 覆盖层。deploy、rollback 与受支持配置更新入口 `update-deploy-env.sh` 必须共用一把严格配置锁，从读取 `.env` 持有到 rollout/回滚验证和最终镜像状态晋升。严格锁存在即返回 75，不得 stale 自动清理；必须人工核对 owner/phase 证据。直接外部写 `.env` 无法加入互斥，明确属于不受支持操作；如在 promotion 前检测到必须保留操作员值并中止。
- current/previous/staged manifest、current/staged image state、summary、history 和备份/验证状态必须使用目标同目录临时文件加原子 rename。
- timeout、失败码、support manifest/锁和信号退出行为守卫必须进入默认 CI；只有文档或静态人工审阅不得称为稳定部署验收。
旧灾备 workflow 直接把 `DEPLOY_ENV_FILE` 重定向到服务器 `.env`，该路径未加入严格配置锁，在迁移到 `update-deploy-env.sh` 并重新验证前不得使用。迁移时候选文件仍必须显式 `chmod 600`，不得只依赖 `umask 077`。
Cloud 可复制运维模板和部署默认值必须使用当前标准目录：Cloud 部署目录为 `/data/iiot-platform/cloud/deploy`，Edge 更新包目录为 `/data/iiot-platform/edge-client/edge-updates`。`deploy/.env.example`、compose fallback、non-root readiness fallback、backup、restore verification 和 post-release cleanup cron 模板不得继续使用旧 `/srv/iiot-cloud/deploy` 或 `/srv/iiot/edge-updates` 示例。

Cloud 发布 preflight 必须拒绝仍指向文档示例仓库的应用镜像或基础设施镜像，例如 `harbor.example.com`、`harbor.internal.example` 或其它 `.example` registry；应用镜像和基础设施镜像都必须带显式 Harbor/内部 registry，不能使用 `iiot-httpapi:sha-*`、`nginx:tag` 这类会隐式走 Docker Hub 或本地命名的镜像引用，也不能等到 `docker compose pull` 阶段才暴露模板值未替换问题。
本机构建、灾备 `cloud-image`、基础镜像 mirror、local release、Edge installer catalog 验收等部署入口也必须拒绝 `.example` / `internal.example` 文档示例域，不能让文档占位值进入 build、push、SSH 或下载验收阶段；这同时适用于 registry、SSH target、Cloud Gateway URL 和 AICopilot challenge URL。
Cloud 本机构建 `REGISTRY`、基础镜像 mirror `MIRROR_REGISTRY` 和灾备 workflow 的 `OCI_REGISTRY` 必须是显式 Harbor/内部 registry host，至少包含 `.`、`:` 或为 `localhost`；不得接受 `harbor` 这类会被 Docker 解释成 Docker Hub namespace 或非显式 registry 的值。`HARBOR_PROJECT`、`MIRROR_NAMESPACE` 和灾备 `OCI_NAMESPACE` 必须是单段小写 Harbor project/namespace，只允许小写字母、数字、`.`、`_`、`-`，不得包含斜杠、空白、反斜杠或 `.example` / `internal.example` 模板值。
Cloud 本机构建只有选中 `web` 或 `--all` 时才要求显式传入 `VITE_AICOPILOT_CHALLENGE_URL`；backend-only 构建不得被 AICopilot 前端 URL 阻断。脚本帮助、README 和 GO_LIVE 中的可复制示例不得使用会被脚本自身拒绝的 `.example` / `internal.example` 值。
Cloud 发布 preflight 必须对生产 secret 做最低强度检查：`JWTSETTINGS__SECRET` 至少 32 字符，数据库、RabbitMQ、Seq 管理员和首管理员密码至少 12 字符；空值、模板值、已知弱值和过短值都必须 fail-fast。
Cloud 发布 preflight 必须校验 Edge 上传限流变量为正整数，且不得超过 nginx `edge_upload_limit` 基线 12000 次/分钟；`RATE_LIMIT_CAPACITY_UPLOAD_*`、`RATE_LIMIT_DEVICE_LOG_UPLOAD_*`、`RATE_LIMIT_PASS_STATION_UPLOAD_*` 不能被设置成 0、非数字或超大值来绕过 HTTP-only 补偿控制。
Cloud 发布 preflight 的 HTTP-only 补偿控制摘要必须反映实际执行过的运行态检查；已有 `current-release.env` 时才打印 `healthz-http-local ops-check-runtime`，干净首部署或 post-clean rebuild 没有旧运行态时必须打印 `runtime-check-skipped-no-current-release`，不得把未执行的旧运行态 healthz/ops-check 说成已通过。
部署文档必须拆成“长期模板/规则”和“当前生产现场口径”两层；当前 Cloud 现场目录、标准 non-root 部署用户、`certs/`/`releases/*` owner/mode、Edge 上传默认值等真实事实，必须同时出现在工作区总入口和 Cloud 部署文档中，不能只藏在脚本默认值、历史复盘或服务器现状里。
如果当前 `Cloud` 与 `AICopilot` 共用同一台生产宿主机，文档还必须同时写清共享宿主机事实、共享标准发布人和两个独立部署根；不得把同机双部署根问题写成两套互不相关的环境。
Cloud 标准 non-root 发布前必须同时验证 `certs/` 与 `releases/current-release*`、`staged-release*`、`previous-release*`、`current-release.summary.md` 对标准部署用户可读可写；root 应急路径一旦写入这些状态，关闭任务前必须恢复 owner/mode 并重新验证标准 non-root preflight。

`deploy/scripts` 中被文档、workflow 或其它脚本以 `./scripts/<name>.sh` / `"$SCRIPT_DIR/<name>.sh"` 直接执行的运维入口脚本，必须在 git 中保持 executable 位；`release-common.sh` 只作为 source 共享库，不作为入口脚本。

关闭 Cloud 容器非 root 生产验收或验收 Edge 安装/更新发布时，不得只依赖 `/internal/healthz`；必须设置 `POST_DEPLOY_VERIFY_EDGE_INSTALLER_CATALOG=1` 运行 `post-deploy-check.sh`，或运行等价的 `verify-edge-installer-catalog.sh`，确认 public catalog、installer artifact、静态 installer 和 Velopack `RELEASES` / `.nupkg` 下载链路可用。Cloud 发布选中 `iiot-httpapi`、`iiot-gateway`、`iiot-web` 或全量发布时，`deploy-release.sh` 必须自动设置 `POST_DEPLOY_REQUIRE_EDGE_INSTALLER_CATALOG=1` 和 `POST_DEPLOY_VERIFY_EDGE_INSTALLER_CATALOG=1`；该场景下 `post_deploy_edge_installer_catalog=skipped` 必须非零退出，不能由操作者口头降级。
Cloud 发布后默认 post-deploy 必须验证 OIDC discovery 和 JWKS 公共端点；OIDC token 发行验证只能通过 `POST_DEPLOY_VERIFY_OIDC_TOKEN=1` 显式打开，并通过 `POST_DEPLOY_OIDC_AUTHORIZATION_CODE_FILE` / `POST_DEPLOY_OIDC_CODE_VERIFIER_FILE` 提供真实 authorization code 和 PKCE verifier。code/verifier 文件必须是可读、非空、仅当前用户可访问的 0600 类私有文件；脚本不得伪造 user/password、client_credentials、OIDC code 或 token，也不得用 discovery/JWKS 结果冒充 token 发行成功；真实 code/verifier 不得进入日志、文档、shell history、进程环境或 curl 进程参数。
