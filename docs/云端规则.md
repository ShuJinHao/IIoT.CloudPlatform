# 云端规则

本文档约束 `IIoT.CloudPlatform`。总规则见 `../../docs/总规则.md`。

## 0. 改动收口门禁

- 修改 `IIoT.CloudPlatform` 代码前，必须先读工作区 `AGENTS.md`、`../../docs/总规则.md`、项目 `AGENTS.md`、本文档、相关专题契约、相关源码、相关测试和近期 git/GitHub 历史。
- 已验收功能默认冻结；不能因为局部重构、测试修复、接口调整或文档整理顺手改变已有业务链路、权限边界、部署口径或数据契约。
- 每次代码改动完成前，必须更新项目滚动复盘文档 `改动复盘与规则沉淀.md`，最新记录放在最前。
- 每次复盘必须写明改动范围、改动原因、影响面、验证命令、验证结果和规则提炼结论。
- 必须判断是否形成长期规则；有则写入本文档、专题契约或工作区长期规则，无则在复盘中写明“无新增长期规则”及原因。
- 最终回复必须列出复盘文档、规则沉淀位置和验证命令；缺任一项，不得称为完成。
- 默认只更新项目滚动复盘文档，不为每个任务新增单独流水文档；只有形成可长期复用的业务或技术契约，才新增专题文档。

## 1. 云端定位

云端是制造业务主数据和长期归档查询的主要来源，负责：

- 人员、账号、身份、权限。
- 设备注册、设备主数据、设备删除约束。
- 配方和配方版本。
- 客户端 bootstrap/寻址。
- 产能、日志、生产数据、过站数据的归档和查询。

## 2. 人员和权限

- 人员入职必须创建账号，人员档案和身份账号是一体业务。
- 人员可先入职，后分配设备机台。
- 客户端修改类操作默认必须同时满足设备分配和功能权限。
- 新设备注册是管理员专属操作，不得放宽为普通业务权限。
- 默认使用角色模板承载常用岗位权限，不要求运维逐个账号手动维护权限点。内置模板包括 `DeviceAdmin`、`ClientInstallerOperator`、`ClientReleaseManager`、`ProductionViewer`、`RoleAdmin`、`HrAdmin`。
- 员工绑定或变更角色属于访问权维护，必须具备 `Employee.UpdateAccess`；只改姓名、启停等基础档案不得隐式获得角色绑定能力。
- `Role.Read` 只允许读取角色名等绑定所需信息；定义角色和修改角色权限仍分别使用 `Role.Define`、`Role.Update`。
- 个人特批权限是例外机制，应优先用角色模板解决常规岗位权限。
- 客户端发布权限必须和设备权限拆开：查看使用 `ClientRelease.Read`，生成客户端下载器使用 `ClientRelease.GenerateInstaller`，发布上传使用 `ClientRelease.Publish`，发布目录和保留策略管理使用 `ClientRelease.Manage`。
- Edge 日常发布不得依赖 Human refresh token 会话；Cloud 必须提供 Edge Release API key 换短期发布 JWT 的机器发布身份。机器发布身份只能携带 `ClientRelease.Read` / `ClientRelease.Publish`，不得获得 `ClientRelease.Manage`、设备权限或管理员绕过。
- Edge Release API key 明文只能创建时返回一次，数据库只保存高熵 key 的 hash；吊销、过期、最后使用时间和创建/吊销操作者必须可审计。

## 3. 设备和身份链

- 设备注册成功后由云端生成唯一 `ClientCode`。
- `ClientCode` 是客户端 bootstrap/寻址码，不是归档主键，不允许维护时手工改写。
- `DeviceId` 是云端正式设备标识，客户端上传产能、日志、生产数据必须使用 bootstrap 后获得的正式身份。
- 设备名称仅用于展示，不得作为上传身份。
- 设备只允许修改名称。
- 设备删除是管理员高危级联硬删除，必须同时满足 `Device.Delete` 和 `Device.CascadeDelete` 权限点。删除设备时必须按 `DeviceId` 显式级联删除配方、产能、设备日志、过站数据、客户端版本快照、客户端运行心跳、上传幂等登记、人员设备授权、设备 refresh token 等关联数据。
- 设备删除前必须展示级联影响范围并二次确认；删除后必须写审计，记录操作者、设备、`ClientCode`、各类删除数量和结果。
- 删除设备前必须先捕获被清理的人员设备授权员工 ID，删除后同步清理这些用户的 `DeviceAccessesByUser` 授权缓存，避免普通用户继续看到旧设备授权。
- 设备删除不得静默删除，不得只依赖数据库隐式 cascade。
- 设备级联删除写路径必须走 EF Core 基础设施服务，并和设备主数据删除、Outbox、审计、缓存失效保持同一业务用例编排；Dapper 只能保留归档读模型和历史记录写入，不得实现设备级联删除命令。对尚未建 EF 实体的历史表，允许在 EF 事务内使用参数化 SQL 删除。

## 4. 配方

- 配方归属于设备和工序。
- 初始版本从 `V1.0` 开始。
- 配方修改必须创建新版本并保留旧版本用于追溯，不得原地覆盖当前版本。

## 5. 过站工序扩展

- 云端普通过站工序采用“公共检索列 + `payload_jsonb` 扩展字段 + 配置驱动 schema”。
- 新增普通过站工序默认只改 `src/hosts/IIoT.HttpApi/config/pass-station-types.json`。
- 不为每个普通工序新增独立 DTO、Event、Mapper、SQL、Consumer、Vue 页面或 RabbitMQ endpoint。
- 统一上传接口：`POST /api/v1/edge/pass-stations/{typeKey}/batch`。
- 统一查询接口：`GET /api/v1/human/pass-stations/{typeKey}` 和 `GET /api/v1/human/pass-stations/{typeKey}/{id}`。
- AI 高频生产数据统一读取接口是 `GET /api/v1/ai/read/production-records`；新工序只要进入 `pass_station_records + pass-station-types.json`，默认具备 AI 通用读取能力，不再为 AI 按工序新增独立查询接口。
- 只有新增公共检索字段、高频 JSONB 查询/统计、全新业务链路、独立权限/审计/状态流转时，才允许突破配置扩展，并且必须单独说明影响范围。

## 5.1 客户端发布和设备状态

- 客户端安装状态只能来自设备版本上报和当前 release catalog 差异；客户端软件状态只能来自运行心跳，不得用版本上报超过 24 小时替代“未运行”判断。
- 运行心跳表必须按 `device_id + client_code` 唯一，保留 `last_heartbeat_at_utc` 索引用于超时查询；心跳上报必须校验 `DeviceId + ClientCode` 身份链。
- `DeviceClientState` 是客户端状态官方投影，按 `DeviceId + ClientCode` 唯一；版本上报和运行心跳进入 Cloud 后必须同步更新该投影。设备客户端状态页、AI Read 和后续状态类查询不得临时拼接 snapshot/heartbeat 重新解释状态。
- 客户端发布运行时聚合根是 `ClientReleaseComponent`，版本集合是 `ClientReleaseVersion`，受控文件清单是 `ClientReleaseArtifact`；宿主和插件不得再维护两套 release 聚合、两套仓储或目录扫描补版本链路。
- 旧 `edge_client_host_releases` / `edge_client_plugin_releases` 只允许出现在历史 migration 和一次性迁移脚本中；运行时查询、发布、首装生成、状态变更、删除和保留策略不得再依赖旧表。
- 管理员物理删除发布包文件必须具备 `ClientRelease.Manage`，并先判断是否在用：只要 `edge_device_client_version_snapshots` 中有设备当前版本等于目标版本，就视为在用，不额外加时间窗口；在用包只能归档或改状态，不得物理删除。
- 发布 catalog、首装生成、Edge 更新 catalog 和公开下载 catalog 的版本来源只能是 Cloud 发布记录；不得在查询时扫描文件目录补出未登记版本，也不得让残留文件把 `Deleted` 包重新显示出来。
- 发布包物理删除只能由服务端 release 元数据和 `EdgeInstallerArtifactOptions.RootPath` 推导受控发布目录下的真实文件路径，不接受前端传入路径；删除前必须限制在受控发布根内，拒绝 `..`、符号链接/重解析点逃逸，并把删除文件清单写入审计日志。
- `Archived` 表示业务归档，文件可能仍存在；`Deleted` 表示已从可分发 catalog 和首装链路移除，发布历史、版本号、更新时间、更新内容、删除原因和失败原因必须保留。

## 6. 架构和代码

- 遵守现有 DDD 分层和依赖倒置。
- 领域规则放领域层，应用编排放服务/用例层，基础设施细节放基础设施层，接口层只收发请求和结果。
- 不把业务判断散落到 Controller、缓存适配层或临时工具类。
- 数据变更影响缓存读取时，必须同步处理缓存失效、刷新或重建。
- 不为未来可能场景提前加入多余抽象、迁移层、兼容层、老接口保留层。
- 开发阶段不保留旧 API、旧 query 参数、旧 DTO 字段、旧路径或双轨兼容；跨项目契约调整必须一次性对齐调用方和被调用方。

## 7. AI 读取边界

- AICopilot 可按批准范围只读云端数据、规则和接口说明。
- 当前没有默认 Cloud 写 API 授权给 AICopilot。
- AI-facing 读取接口只使用 Cloud 正式业务身份，不接受调用方把展示名、设备码或临时参数伪装成正式主键。
- AI 高频业务读取必须通过 Cloud `/api/v1/ai/read/*` 只读 API，不让 AI 端直接连生产库；已开放的数据域包括设备、工序、客户端发布版本、设备客户端状态、产能汇总、小时产能、设备日志和通用生产数据。
- AI 通用生产数据只使用 `/api/v1/ai/read/production-records`，返回公共列和 schema 化字段；不得给 AI 返回 raw `payload_jsonb` 字符串，也不得继续暴露 AI 专用 `/pass-stations/{typeKey}` 语义入口。
- `DeviceId` 是 AI 读取产能、日志和过站记录的正式设备参数；`ClientCode`/`deviceCode` 只可用于设备查询或 bootstrap 寻址，不能冒充 `deviceId`。
- 当前 AICopilot 规则禁止读取未批准的配方主数据和配方版本；Cloud 不应为 AI 侧保留未被当前目标使用的配方读取兼容入口。
- Text-to-SQL 只作为低频探索兜底，不能替代已沉淀的日志、产能和生产数据 AI Read API；Text-to-SQL 必须只读、限行、限时、审计并限制表范围。
- 若未来提供新的 AI-facing API，必须先明确契约、权限、审计、幂等、允许动作和禁止动作。

## 8. 验证

常用命令：

```bash
dotnet build IIoT.CloudPlatform/src/hosts/IIoT.AppHost
dotnet test IIoT.CloudPlatform/src/tests/IIoT.EndToEndTests
dotnet test IIoT.CloudPlatform/src/tests/IIoT.ServiceLayer.Tests
dotnet test IIoT.CloudPlatform/src/tests/IIoT.ProductionService.Tests
cd IIoT.CloudPlatform/src/ui/iiot-web && npm run build
```

GitHub `cloud-ci` 的 push/PR 默认是快速门禁：restore/build、ServiceLayer、ConfigurationGuard、前端 build、compose config。完整 `IIoT.EndToEndTests` 只在手动 `workflow_dispatch` 勾选时运行，避免每次部署被重型集成测试阻塞。

Cloud 生产发布标准路径是本机构建镜像、推送 Harbor、再通过 SSH 触发服务器 `deploy/scripts/deploy-release.sh`；`cloud-image` / `cloud-deploy` 只允许作为带确认词的灾备入口。单镜像 build/push 默认 15 分钟超时，Harbor 检查默认 2 分钟超时，SSH deploy 默认 40 分钟超时；超时必须停止并诊断 Docker buildx、Harbor tag、服务器 compose/logs 和 release 状态，不得继续等待 GitHub Actions 或 shell 命令自然超时。

新增或调整过站工序配置后，至少验证后端构建、服务层测试和前端构建。
