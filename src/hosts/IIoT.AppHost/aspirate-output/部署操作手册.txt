# IIoT Cloud Platform 部署操作手册

本文档只说明当前云端部署模板与脚本的标准用法。

当前入口拓扑：

- `Nginx`
- `IIoT.Gateway`
- `IIoT.HttpApi`

当前部署策略：

- 单 IP
- HTTP
- 不启用正式 TLS/证书

## 一、部署前准备

1. 进入 `src/hosts/IIoT.AppHost/aspirate-output/`
2. 复制一份配置样例：

```powershell
Copy-Item .env.example .env
```

3. 在 `.env` 中至少填写这些键：

- `IIOT_REGISTRY`
- `DEPLOY_HOST`
- `DEPLOY_USER`
- `DEPLOY_PORT`
- `STACK_NAME`
- `DEPLOY_DIR`
- `PUBLIC_BASE_URL`
- `PG_PASSWORD`
- `RABBITMQ_DEFAULT_USER`
- `RABBITMQ_DEFAULT_PASS`
- `ASPIRE_DASHBOARD_FRONTEND_BROWSERTOKEN`
- `ASPIRE_DASHBOARD_OTLP_PRIMARYAPIKEY`

4. `PUBLIC_BASE_URL` 使用正式外部访问地址，例如：

```text
PUBLIC_BASE_URL=http://10.0.0.15
```

注意：不要以 `/` 结尾。

## 二、本地构建与推送

在 `aspirate-output` 目录执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\build-push.ps1
```

如果这次只改了部署模板或配置文件，没有改镜像内容：

```powershell
powershell -ExecutionPolicy Bypass -File .\build-push.ps1 -SkipBuild
```

脚本按这个优先级取配置：

1. 脚本参数
2. 当前目录 `.env`
3. 进程环境变量

## 三、部署到服务器

推荐直接使用脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\deploy.ps1
```

如果镜像已经提前推送完成，可跳过构建：

```powershell
powershell -ExecutionPolicy Bypass -File .\deploy.ps1 -SkipBuild
```

`deploy.ps1` 会完成这些动作：

1. 登录 `IIOT_REGISTRY`
2. 构建并推送镜像（可跳过）
3. 将 `.env`、`docker-compose.yaml`、`nginx.conf` 上传到 `${DEPLOY_USER}@${DEPLOY_HOST}:${DEPLOY_DIR}`
4. 执行 `docker stack deploy`

## 四、验证部署结果

查看所有服务状态：

```powershell
docker stack ps ${STACK_NAME}
```

当前常见服务包括：

```text
iiot-httpapi
iiot-gateway
iiot-web
iiot-dataworker
iiot-migration
nginx-gateway
postgres
redis-cache
rabbitmq
aspire-dashboard
```

查看 HttpApi 日志：

```powershell
docker service logs ${STACK_NAME}_iiot-httpapi --tail 50
```

查看 Gateway 日志：

```powershell
docker service logs ${STACK_NAME}_iiot-gateway --tail 50
```

## 五、常用访问地址

- 前端地址：`${PUBLIC_BASE_URL}`
- API 地址：`${PUBLIC_BASE_URL}/api`
- Aspire Dashboard：`http://${DEPLOY_HOST}:18888`
- RabbitMQ 管理面板：`http://${DEPLOY_HOST}:15672`

RabbitMQ 管理面板请使用 `.env` 中的 `RABBITMQ_DEFAULT_USER` / `RABBITMQ_DEFAULT_PASS` 登录，不要使用示例值。

## 六、常用运维命令

查看 Stack 服务：

```powershell
docker stack services ${STACK_NAME}
```

强制重启某个服务：

```powershell
docker service update --force ${STACK_NAME}_iiot-httpapi
docker service update --force ${STACK_NAME}_iiot-gateway
```

扩缩容：

```powershell
docker service scale ${STACK_NAME}_iiot-httpapi=3
```

移除整个 Stack（慎用）：

```powershell
docker stack rm ${STACK_NAME}
```

## 七、2026-04-22 deployment fix notes

The current `aspirate-output` template must include these runtime values before upload:

- `JWTSETTINGS__SECRET`
- `SEED_ADMIN_NO`
- `SEED_ADMIN_PASSWORD`
- `SEED_ADMIN_REAL_NAME` (optional, defaults to `系统管理员`)

`iiot-migration` is a one-shot initialization service.
When database migration and seed succeed, `docker service ls` may show `${STACK_NAME}_iiot-migration` as `0/1`. This is expected.

If you need a full reset of the database before redeploy, run:

```powershell
docker stack rm ${STACK_NAME}
docker volume rm ${STACK_NAME}_postgres-iiot
```

## 2026-04-22 deployment guardrails

This section overrides any earlier emergency-reset wording for normal release work.

- Routine deploy and routine update must preserve the existing PostgreSQL data volume.
- Do not delete the database, do not remove the PostgreSQL volume, and do not use full reset steps during normal deployment/update.
- Only perform `docker stack rm`, `docker volume rm`, or any database rebuild when the user explicitly asks for a full reset and understands the data-loss impact.
- Before upload/deploy, verify every `${VAR}` referenced by `docker-compose.yaml` exists in both `.env` and `.env.example`.
- Keep `GATEWAY_HTTP_PORT=81` unless the user explicitly approves a port change.
- `iiot-migration` must receive `JWTSETTINGS__SECRET`, `SEED_ADMIN_NO`, `SEED_ADMIN_PASSWORD`, and `SEED_ADMIN_REAL_NAME`.
- `iiot-web` updates must be verified end-to-end: no CSP-blocked external font/CDN dependency, and the container must really serve `dist` on the port expected by `nginx.conf`.

## 八、2026-04-27 限流基线

当前限流改为两层分工：

- `nginx` 只做入口粗防护。
- `IIoT.HttpApi` 做业务级精细限流。

入口层要求：

- 登录、刷新、bootstrap 继续在 `nginx` 保护。
- 边端上传接口在 `nginx` 只保留宽松保护，避免多台设备共用一个出口 IP 时互相误伤。
- `refresh` 必须单独有自己的 `location` 和 `refresh_limit`，不得继续和密码登录共用同一条入口限流。

应用层要求：

- `password-login`、`refresh`、`edge-operator-login`、`bootstrap` 必须分开配置。
- `capacity-upload`、`device-log-upload`、`pass-station-upload` 必须分开配置。
- 上传限流必须优先按 `DeviceId` 分区；只有拿不到 `DeviceId` 时才允许退回按 IP。

后续新增工序时：

- 如果是新的高频云端上传接口，默认先归入 `pass-station-upload` 这一类思路评估，不得直接复用旧的单一 `edge-upload` 概念。
- 如果只改了阈值，也必须同时检查两份 `nginx.conf` 和 `IIoT.HttpApi/appsettings.json`，避免入口层与应用层配置漂移。
