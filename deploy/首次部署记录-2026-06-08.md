OIDC 签名证书是 Cloud 本地 token 签名证书，不是公网 HTTPS 证书，不需要购买。服务器首次启用 OIDC 前生成一次：

```bash

openssl req -x509 -newkey rsa:2048 \
  -keyout /tmp/cloud-oidc-signing.key \
  -out /tmp/cloud-oidc-signing.crt \
  -days 3650 -nodes \
  -subj "/CN=IIoT Cloud OIDC"
openssl pkcs12 -export \
  -out /srv/iiot-cloud/deploy/certs/cloud-oidc-signing.pfx \
  -inkey /tmp/cloud-oidc-signing.key \
  -in /tmp/cloud-oidc-signing.crt \
  -passout pass:
rm -f /tmp/cloud-oidc-signing.key /tmp/cloud-oidc-signing.crt
sudo chmod 600 /srv/iiot-cloud/deploy/certs/cloud-oidc-signing.pfx
```

## 4. 本次现场修正

这次服务器上做过两个现场修正，后续应该固化回仓库。

### 4.1 nginx 内部健康检查

`/internal/healthz` 原来只允许 `127.0.0.1` 和 `::1`，但宿主机通过端口访问容器时，容器看到的来源是 Docker 网桥地址，例如 `172.19.0.1`。因此需要允许 Docker bridge 网段：

```nginx
location = /internal/healthz {
    allow 127.0.0.1;
    allow ::1;
    allow 172.16.0.0/12;
    deny all;
    proxy_pass http://gateway_pool;
}
```

改完必须重启 nginx 容器：

```bash
docker compose --env-file ./.env -f docker-compose.prod.yml restart nginx-gateway
```

### 4.2 DataWorker Docker healthcheck

DataWorker 主进程和手工 `docker exec` healthcheck 都可以通过，但镜像 Dockerfile 自带的 `CMD-SHELL` healthcheck 曾经在 Docker 健康状态里持续报 `missing connection string 'iiot-db'`。

服务器上用 compose 覆盖成 exec form 后恢复：

```yaml
iiot-dataworker:
  healthcheck:
    test: ["CMD", "dotnet", "IIoT.DataWorker.dll", "--healthcheck"]
    interval: 30s
    timeout: 15s
    retries: 3
    start_period: 20s
```

重建 DataWorker：

```bash
docker compose --env-file ./.env -f docker-compose.prod.yml up -d --force-recreate iiot-dataworker
```

最终状态应为：

```text
deploy-iiot-dataworker-1  Up ... (healthy)
```

### 4.3 HTTP 临时模式

当前外部访问使用 `http://10.98.90.154:81`。正式生产应该上 TLS 后回到 Production/OIDC 正常配置。若继续用 HTTP，OIDC issuer 校验会拒绝非 HTTPS，需要临时降级环境配置；这只能算首次连通临时方案，不是正式生产安全形态。

### 4.4 管理员播种修复

首次部署后登录页提示“工号或密码不正确”，不是前端问题。现场确认数据库状态如下：

- `AspNetRoles` 有 `Admin` 角色。
- `AspNetUsers` 为 `0` 行。
- `employees` 为 `0` 行。

也就是说迁移/播种没有创建管理员账号，导致任何工号都无法登录。现场已把服务器 `/srv/iiot-cloud/deploy/.env` 收敛为：

```env
SEED_ADMIN_NO=101650
SEED_ADMIN_PASSWORD=<用户指定的生产密码，仅保存在服务器 .env>
SEED_ADMIN_REAL_NAME=系统管理员
```

并在数据库中补建：

- `AspNetUsers.UserName = 101650`
- `AspNetUserRoles = Admin`
- `employees.employee_no = 101650`
- `employees.is_active = true`

## 6. 这次低效点和下次规则

这次低效的根因不是部署本身复杂，而是把“首次部署”按“兼容升级”思路处理了。下次按下面规则执行：

1. 首次部署已确认可清数据时，第一步直接 `down -v --remove-orphans`，不用保护旧库、旧卷、旧应用容器。
2. Harbor 和应用网关不能抢同一个端口：Harbor 保持 `80`，应用网关用 `81`。
3. 服务器拉 Harbor 内镜像时优先使用 `127.0.0.1:80`，避免绕外部地址。
4. `deploy-release.sh` 当前更像升级脚本，首次部署前备份和备份验证会拖慢；后续应加 `FIRST_DEPLOY=1` 或单独 `first-deploy.sh`。
5. 健康检查必须看 Docker 最终状态，不能只看手工 `docker exec` 通过。
6. 首次部署完成后必须立刻验证管理员登录；只看首页 `200` 不够。
7. 现场修正必须回写仓库，否则下次新服务器还会重复踩坑。

## 7. 后续应固化

- 把 nginx `/internal/healthz` 的 Docker bridge 放行写回 `deploy/nginx/nginx.conf`。
